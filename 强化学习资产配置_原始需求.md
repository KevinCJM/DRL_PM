## 2. 平台算法范围

平台必须支持以下算法类别。

### 2.1 传统资产配置与规则型策略

必须实现：

```text
fixed_ratio                         固定比例策略
equal_weight                        等权重策略
buy_and_hold                        买入持有策略
traditional_markowitz_mean_variance 传统马科维兹均值方差策略
markowitz_min_variance              最小方差策略
markowitz_max_sharpe                最大夏普策略
risk_parity                         风险平价策略
inverse_volatility                  逆波动率策略
minimum_drawdown                    最小回撤策略
risk_evaluation                     风险评价策略
HRP                                 Hierarchical Risk Parity 策略
momentum                            动量 / 趋势策略
```

所有传统策略必须接入统一的再平衡调度器 `RebalanceScheduler`，并使用统一的交易成本模型、约束管理器和回测引擎。

### 2.2 深度学习与强化学习基准模型

必须实现：

```text
PPO                                  PPO 连续权重策略
CNN-PPO                              CNN 编码器 + PPO 连续权重策略
Bernoulli-Gated CNN-PPO              Bernoulli gate + CNN-PPO
DQN-Only Template Selector           DQN 离散组合模板选择器
EIIE                                 Ensemble of Identical Independent Evaluators 风格组合模型
Full DQN-Gated Multi-Task CNN-PPO    DQN 调仓门控 + PPO 连续权重 + 多任务表征学习
```

### 2.3 平台强化学习算法模块

必须实现以下 5 类强化学习算法模块：

```text
A. Full DQN-Gated Multi-Task CNN-PPO
B. Preference-Conditioned DQN-Gated PPO
C. Uncertainty-Aware DQN-Gated PPO
D. Distributional-CVaR DQN-Gated PPO
E. Partial-Rebalance DQN / Beta-Gated PPO
```

这 5 类算法模块需要共享数据管线、环境、成本模型、约束模块、评估器、结果输出 schema 和训练框架。

### 4.1A 数据治理协议

所有实验必须显式记录以下数据治理口径：

```yaml
data_governance:
  corporate_action_policy: adjusted_close_or_nav_first
  model_return_feature_field: log_return
  execution_return_mode: derived_from_execution_prices
  execution_price_fields:
    close: wide_close
    open: wide_open
  close_to_close_return_field: wide_pct_chg
  log_return_to_simple_return_allowed: true
  dividend_policy: reflected_in_adjusted_price_or_nav
  calendar_id: cn_a_share_etf_daily
  benchmark_series:
    default: equal_weight_available_assets
    optional:
      - CSI300
      - CSI500
      - SSE50
      - custom_csv
  risk_free_rate:
    mode: constant_annual
    annual_value: 0.015
    optional_series_file: null
  timezone: Asia/Shanghai
```

实现规则：

```text
1. model_return_feature_field 用于模型输入、辅助标签和诊断。
2. execution_return_mode = derived_from_execution_prices 时，NAV 更新必须由 execution_price_fields 中的 open / close 价格推导实际持有区间收益。
3. close_to_close_return_field 只可直接支持 same_close_idealized_execution_enabled 或明确的 close-to-close 口径。
4. 若 execution_price = next_open，return_from_decision_to_execution = open_{t+1} / close_t - 1，holding_simple_return = close_{t+1} / open_{t+1} - 1，不得直接用 wide_pct_chg 代替 holding_simple_return。
5. 若 same_close_idealized_execution_enabled = true，可使用 wide_pct_chg 作为 holding_simple_return。
6. wide_log_return 仅用于模型输入、辅助标签、诊断日志或由 NAV 反推的指标校验。
7. 若同时存在价格收益与净值收益，优先使用已复权或净值口径的收益字段。
8. 所有收益、benchmark、risk-free 必须对齐到同一交易日历。
9. 资产在某日不可用时 availability_mask = false；该资产不得参与组合收益、归一化、风险估计、PCA fit 或横截面标准化。
10. 必须支持 availability_reason 字段，枚举为 listed、not_listed_yet、delisted、suspended、missing_quote、missing_return。
11. not_listed_yet 不允许建仓；suspended 已有持仓冻结；delisted 触发强制退出或现金替代；missing_quote / missing_return 必须记录 warning。
12. 若 data/processed 或 data/metrics_factory 缺失，loader 必须 fail fast，并报告缺失文件清单；不得联网自动补数。
13. 若启用 akshare_etf_fetcher 重新生成数据，必须由显式命令触发，且生成新的 data_manifest。
```

MetricsFactory 全量审计规则：

```text
1. all_metrics_features.parquet 中除 date / ts_code 外的全部列先进入候选集。
2. FeatureMatrixBuilder 必须对 bundle.metrics_features 全量列执行 provenance + leakage audit。
3. pass 与 warning 列允许进入 G3_metrics_factory 模型输入。
4. fail 列不得进入模型输入，但必须出现在 feature_provenance_report.csv。
5. dropped 表示非泄漏原因删除，例如输入矩阵未启用、特征选择删除或显式配置删除。
6. bundle.feature_cols 不能替代审计，只能作为审计后的入模白名单或已选择特征列表。
```

MetricsFactory 泄漏状态：

```text
pass:
  已确认只使用 feature date 及以前历史窗口。

warning:
  无法完全解析来源或窗口，但列名未命中未来标签黑名单；默认保留并记录。

fail:
  列名命中 future / forward / label / target / next / lead / t+1。
  或 manifest 标明使用未来数据、全样本统计。
  或无法确认历史窗口截至 feature date。

dropped:
  审计后因矩阵配置、特征选择、PCA 前筛选或显式黑名单被删除。

feature_audit.warning_policy:
  keep | drop | report_only
  默认 keep。
  keep 表示 warning 特征可进入模型并写入报告。
  drop 表示 warning 特征不得进入模型。
  report_only 表示 warning 特征按配置入模，但必须输出单独诊断汇总。
```

特征组：


G2_basic_technical 固定公式：

```text
technical_windows = [5, 10, 20, 60, 120]

moving average:
  close_ma_ratio_w        = close / rolling_mean(close, w) - 1
  ma_5_over_20_ratio      = rolling_mean(close, 5) / rolling_mean(close, 20) - 1
  ma_20_over_60_ratio     = rolling_mean(close, 20) / rolling_mean(close, 60) - 1

momentum:
  momentum_log_return_w   = log(close_t / close_{t-w})
  rolling_log_return_sum_w = rolling_sum(log_return, w)

volatility:
  rolling_volatility_w    = rolling_std(pct_chg, w)
  rolling_downside_vol_w  = rolling_std(min(pct_chg, 0), w)

drawdown:
  drawdown_signed_w       = close / rolling_max(close, w) - 1
  drawdown_abs_w          = max(0, 1 - close / rolling_max(close, w))
  max_drawdown_abs_w      = rolling_max(drawdown_abs_w over w)

liquidity / volume:
  log1p_amount            = log(1 + max(amount, 0))
  log1p_vol               = log(1 + max(vol, 0))
  amount_ratio_20         = amount / rolling_mean(amount, 20) - 1
  vol_ratio_20            = vol / rolling_mean(vol, 20) - 1
  turnover_rate           = turnover_rate
  turnover_rate_ma_20     = rolling_mean(turnover_rate, 20)

intraday bar shape:
  high_low_over_close     = (high - low) / close
  close_open_over_open    = (close - open) / open
```

## 7. 再平衡调度器

必须实现统一再平衡调度器：


支持模式：

```text

daily                         每个交易日再平衡
weekly                        每周首个或最后一个交易日再平衡
monthly                       每月首个或最后一个交易日再平衡
quarterly                     每季度首个或最后一个交易日再平衡

every_n_days                  每 n 个交易日再平衡
calendar_dates                指定日期列表再平衡

threshold_turnover            目标权重与当前权重差异超过阈值再平衡
volatility_event              波动率超过阈值再平衡
```

调度模式分层：

```text
无需候选权重的模式：
  never
  once
  daily
  weekly
  monthly
  quarterly
  yearly
  every_n_days
  calendar_dates
  volatility_event
  drawdown_event
  risk_budget_breach

需要候选权重的模式：
  threshold_weight_drift
  threshold_turnover

需要候选权重的模式必须在 strategy 生成 candidate_weights 后执行 post_check。
若 candidate_weights 为 None，threshold_weight_drift 与 threshold_turnover 不得返回 true。
```

RebalanceScheduler 与 DQN gate 的权限关系：

```text
1. RebalanceScheduler 决定当前日期是否允许考虑调仓。
2. DQN gate 决定在允许调仓的日期是否执行调仓。
3. 最终调仓动作定义为：

   final_rebalance_action_t =
       1 if scheduler_allowed_t == true and gate_action_t == 1
       else 0

4. 若 rebalance.mode = daily，则 DQN gate 每个交易日都有机会决策。
5. 若 rebalance.mode = monthly，则 DQN gate 只在月度调仓日决策。
6. 非调仓允许日，DQN gate 可跳过计算，也可仅记录诊断值，但不得执行交易。
```

再平衡执行逻辑：

```text
1. decision_weights_t 为 decision_date close 后策略可见的当前组合权重。
2. RebalanceScheduler.pre_check 判断当前日期是否允许考虑调仓。
3. 策略在允许窗口内生成 candidate_weights；需要候选权重的调度模式继续执行 post_check。
4. 若策略包含 DQN gate，则 final_rebalance_action_t = scheduler_allowed_t AND gate_action_t。
5. PortfolioExecutionCore 根据 return_from_decision_to_execution 计算 pre_execution_drifted_weights_t。
6. 如果不执行调仓：
     executed_weights = pre_execution_drifted_weights_t
     realized_turnover = 0
     realized_cost = 0
7. 如果执行调仓：
     PortfolioExecutionCore 使用 tradeable_mask_at_execution 二次检查并修正 target_weights。
     CostModel 根据 pre_execution_drifted_weights_t、target_weights、execution_market_state、portfolio_state 计算 realized_cost。
     executed_weights = 修正后的 target_weights
8. 组合收益与 NAV 使用两阶段 NAV 公式计算。
```

所有传统策略、深度学习策略、强化学习策略都必须通过同一个再平衡与回测接口执行。

```text

    dot(decision_weights_t, execution_market_state.return_from_decision_to_execution)

    drift(decision_weights_t, execution_market_state.return_from_decision_to_execution)

turnover_t =
    0.5 * sum_i |target_weights_i,t - pre_execution_drifted_weights_i,t|

transaction_cost_t =
    CostModel(pre_execution_drifted_weights_t, target_weights_t, execution_market_state, portfolio_state)

NAV_after_cost_t = NAV_execution_t * (1 - transaction_cost_t)

post_execution_return_t =
    dot(executed_weights_t, execution_market_state.holding_simple_return)

NAV_next_t = NAV_after_cost_t * (1 + post_execution_return_t)

    (1 + pre_execution_return_t) * (1 + post_execution_return_t) - 1
```

```text

return_from_decision_to_execution 表示决策后、成交前旧持仓承担的资产 simple return。
```

## 11. 约束管理器

必须实现：

```python
class ConstraintManager:
    def project(
        self,
        raw_weights,
        available_mask,
        portfolio_state,
        decision_market_state,
        reference_weights: np.ndarray | None = None,
    ) -> ConstraintResult: ...
```

支持约束：

```text
long-only
sum-to-one over available assets
unavailable weight = 0
max_weight
min_weight
turnover_limit
drawdown_limit
volatility_limit
CVaR_limit
HHI concentration limit
asset-class exposure limit if metadata exists
cash / money-market lower bound if configured
```

约束处理方式：

```text
hard projection
soft penalty
PPO-Lagrangian
```

投影基础问题：

```text
minimize ||w - raw_w||_2^2
subject to:
  sum_i w_i = 1 over available assets
  w_i >= 0
  w_i = 0 if unavailable_i == 0
  w_i <= max_weight_i
  turnover(w, pre_execution_drifted_weights) <= turnover_limit if configured
  HHI(w) <= HHI_limit if configured
```

`ConstraintResult` 必须包含：

```text
projected_weights
projection_distance
constraint_violations
active_constraints
```

`reference_weights` 规则：

```text
reference_weights 用于 turnover_limit 约束。
decision 阶段默认 reference_weights = decision_weights_t。
execution 阶段必须使用 reference_weights = pre_execution_drifted_weights_t。
若配置了 turnover_limit 且 reference_weights is None，ConstraintManager 必须 fail fast。
```

约束冲突优先级：

```text
1. availability_mask
2. long-only
3. simplex sum-to-one over available assets
4. max_weight
5. min_weight
6. turnover_limit
7. HHI concentration limit
8. CVaR / drawdown / volatility soft limits
9. asset-class exposure limits
```

不可行兜底：

```text
1. ConstraintManager 必须先运行 feasibility check，再执行 projection。
2. 若 available assets 为空，进入 cash 或 money_market；若 cash_enabled=false 且无 money_market，直接报错 NO_AVAILABLE_ASSET。
3. 若 max_weight * n_available < 1，放宽 max_weight 到 1 / n_available，并记录 violation。
4. 若 min_weight * n_available > 1，放宽 min_weight 到 0，并记录 violation。
5. unavailable asset 的 target_weight 永远优先为 0，优先级高于 min_weight。
6. HHI 的理论下限为 1 / n_available；若 HHI_limit < 1 / n_available，放宽到 1 / n_available，并记录 violation。
7. max_weight、min_weight、HHI、simplex 同时冲突时，优先保证 availability、long-only、simplex，再放宽低优先级约束。
8. turnover_limit 与 rebalance 冲突时，投影到距离 raw_weights 最近且满足 turnover_limit 的组合；不得直接拒绝环境 step。
9. 软风险约束默认只进入 reward penalty 和 violation report，不得破坏 simplex。
10. 所有被放宽的约束必须写入 active_constraints 与 constraint_violations。
```

### 13.2 固定比例策略 `fixed_ratio`

用途：

```text
使用预设静态权重进行资产配置，并按再平衡调度器恢复到目标比例。
```

目标权重来源：

```text
1. 直接配置每个 ts_code 的权重；或
2. 配置资产类别权重，再在类别内部等权；或
3. 配置资产类别权重，再在类别内部按逆波动率分配。
```

算法逻辑：

```text
1. 读取 fixed_ratio_weights。
2. 根据 available_mask 将不可用资产权重置为 0。
3. 对可用资产权重重新归一化。
4. 如果 RebalanceScheduler 返回 False，则保持 drifted_weights。
5. 如果 RebalanceScheduler 返回 True，则调仓到归一化后的 fixed_ratio_weights。
6. 记录 turnover、transaction_cost、executed_weights。
```

默认配置示例：

```yaml

  asset_class_weights:
    equity: 0.35
    bond: 0.20
    money_market: 0.10
    overseas_equity: 0.20
    commodity: 0.15
  intra_bucket_mode: equal_weight

    mode: monthly
```

默认资产类别映射：

```text

bond:
  国债 ETF、政策性金融债 ETF、信用债 ETF。

money_market:
  货币 ETF、现金管理类 ETF。
```

### 13.5 马科维兹均值方差策略 `traditional_markowitz_mean_variance`

用途：

```text
在历史窗口内估计期望收益和协方差，求解风险收益折中组合。
```

优化目标：

```text
maximize mu^T w - lambda_risk * w^T Sigma w
```

约束：

```text
sum_i w_i = 1
w_i >= 0
w_i <= max_weight
w_i = 0 if unavailable_i == 0
```

算法逻辑：

```text
1. 在每个再平衡日取过去 lookback_window 个交易日收益。
2. 只使用当前可用资产。
3. 估计 mu。
4. 估计 Sigma。
5. 可选使用 Ledoit-Wolf 或 shrinkage 协方差。
6. 求解二次规划。
7. 对结果执行 ConstraintManager。
8. 非再平衡日保持 drifted_weights。
```

参数：

```yaml
markowitz_mean_variance:
  lookback_window: 252
  expected_return_estimator: historical_mean
  covariance_estimator: ledoit_wolf
  lambda_risk: 10.0
  max_weight: 0.30
  rebalance:
    mode: monthly
```

fallback：

```text
如果有效资产数不足、协方差不可逆或优化失败，则使用 equal_weight 或 inverse_volatility。
```

优化目标：


约束：

```text
sum_i w_i = 1
```

算法逻辑：

```text
1. 在再平衡日估计历史协方差 Sigma。
2. 求解最小方差权重。
3. 通过 ConstraintManager 修正权重。
4. 计算与 drifted_weights 的 turnover 和 cost。
```

### 13.7 最大夏普策略 `markowitz_max_sharpe`

用途：

```text
最大化历史估计期望收益相对于风险的比率。
```

优化目标：

```text
maximize (mu^T w - risk_free_rate) / sqrt(w^T Sigma w)
```

等价实现方式：

```text
1. 可直接用非线性优化器求解；或
2. 用网格 / 多起点 SLSQP；或
3. 转换为风险约束下最大收益问题。
```

算法逻辑：

```text
1. 在再平衡日估计 mu 和 Sigma。
2. 根据 risk_free_rate 计算目标函数。
3. 求解 long-only 最大夏普组合。
4. 对结果执行约束修正。
5. 非再平衡日保持 drifted_weights。
```

参数：

```yaml

  covariance_estimator: ledoit_wolf

  rebalance:
    mode: monthly
```

### 13.8 风险平价策略 `risk_parity`

用途：

```text
让各资产对组合总风险的贡献尽量相等。
```

组合波动率：

```text
sigma_p = sqrt(w^T Sigma w)
```

边际风险贡献：

```text
MRC_i = (Sigma w)_i / sigma_p
```

总风险贡献：

```text
RC_i = w_i * MRC_i
```

目标：

```text
RC_i ≈ sigma_p / n
```

优化目标：

```text
minimize sum_i (RC_i - target_RC_i)^2
```

算法逻辑：

```text
1. 在再平衡日估计协方差 Sigma。
2. 只对可用资产求解风险平价权重。
3. 支持等风险预算和自定义风险预算。
4. 执行 long-only、sum-to-one、max_weight 约束。
5. 非再平衡日保持 drifted_weights。
```

配置：

```yaml
risk_parity:
  lookback_window: 252
  covariance_estimator: ledoit_wolf
  risk_budget: equal
  max_weight: 0.30
  rebalance:
    mode: monthly
```

### 13.9 逆波动率策略 `inverse_volatility`

用途：

```text
根据历史波动率的倒数分配权重。
```

公式：

```text
raw_w_i = 1 / (vol_i + eps)
w_i = raw_w_i / sum_j raw_w_j
```

算法逻辑：

```text
1. 在再平衡日计算每个可用资产过去 lookback_window 的年化波动率。
2. 计算逆波动率权重。
3. 对不可用资产设置 0。
4. 对可用资产归一化。
5. 通过 ConstraintManager 修正。
6. 非再平衡日保持 drifted_weights。
```

配置：

```yaml
inverse_volatility:
  lookback_window: 252
  volatility_floor: 1.0e-6
  max_weight: 0.30
  rebalance:
    mode: monthly
```

### 13.10 最小回撤策略 `minimum_drawdown`

用途：

```text
在历史窗口内选择最大回撤较低的组合。
```

历史组合净值：

```text
NAV_tau(w) = prod_{k=start}^{tau} (1 + r_k^T w)
```

历史最大回撤：

```text
MDD(w) = max_tau (1 - NAV_tau(w) / max_{u <= tau} NAV_u(w))
```

优化目标：

```text
minimize MDD(w) + lambda_vol * volatility(w) - lambda_ret * mean_return(w)
```

算法逻辑：

```text
1. 在再平衡日取历史收益矩阵。
2. 通过 SLSQP、坐标搜索或随机权重采样求解低回撤组合。
3. 对结果执行 long-only、sum-to-one、max_weight 约束。
4. 非再平衡日保持 drifted_weights。
```

配置：

```yaml
minimum_drawdown:
  lookback_window: 252
  optimizer: random_search_then_slsqp
  n_random_portfolios: 5000
  lambda_vol: 0.10
  lambda_ret: 0.05
  max_weight: 0.30
  rebalance:
    mode: monthly
```

### 13.11 风险评价策略 `risk_evaluation`

用途：

```text
基于收益、波动、回撤、下行风险、流动性等指标构造资产风险评分，再根据评分分配权重。
```

资产评分：

```text
score_i =
    a_return * z(return_i)
  - a_vol * z(volatility_i)
  - a_mdd * z(max_drawdown_i)
  - a_downside * z(downside_vol_i)
  - a_cvar * z(CVaR_loss_i)
  + a_liquidity * z(liquidity_i)
```

权重生成：

```text
positive_score_i = max(score_i, 0)
if sum positive_score_i > 0:
    w_i = positive_score_i / sum positive_score_i
else:
    fallback to inverse_volatility or equal_weight
```

算法逻辑：

```text
1. 在再平衡日计算每个资产的滚动收益、波动、回撤、下行波动、CVaR、流动性。
2. 对每个指标在横截面上做 winsorize 和 z-score。
3. 根据配置权重合成 score。
4. 根据 positive score 转换为目标权重。
5. 通过 ConstraintManager 修正。
6. 非再平衡日保持 drifted_weights。
```

配置：

```yaml
risk_evaluation:
  lookback_window: 252
  score_weights:
    return: 0.30
    volatility: 0.20
    max_drawdown: 0.20
    downside_volatility: 0.15
    cvar_loss: 0.10
    liquidity: 0.05
  score_transform: positive_clip
  fallback: inverse_volatility
  max_weight: 0.30
  rebalance:
    mode: monthly
```

### 13.12 HRP 策略 `HRP`

用途：

```text
基于资产相关性聚类和递归二分构造层次化风险平价组合。
```

算法逻辑：

```text
1. 在再平衡日取过去 lookback_window 的收益矩阵。
2. 估计相关矩阵 Corr。
3. 计算距离矩阵 distance_ij = sqrt(0.5 * (1 - Corr_ij))。
4. 对资产做层次聚类。
5. 对聚类结果做 quasi-diagonalization。
6. 递归二分分配权重。
7. 对结果执行 ConstraintManager。
8. 非再平衡日保持 drifted_weights。
```

配置：

```yaml
HRP:
  lookback_window: 252
  linkage_method: single
  covariance_estimator: ledoit_wolf
  max_weight: 0.30
  rebalance:
    mode: monthly
```

### 13.13 动量策略 `momentum`

用途：

```text
基于历史收益或趋势强度选择资产并分配权重。
```

动量指标：

```text
momentum_i = log(close_t / close_{t-lookback})
```

可选风险调整动量：

```text
risk_adjusted_momentum_i = momentum_i / (vol_i + eps)
```

算法逻辑：

```text
1. 在再平衡日计算每个可用资产的动量指标。
2. 按指标排序。
3. 选择 top_k 或 score > threshold 的资产。
4. 权重可采用等权、逆波动率或 softmax(score)。
5. 其他资产权重为 0。
6. 通过 ConstraintManager 修正。
7. 非再平衡日保持 drifted_weights。
```

配置：

```yaml
momentum:
  lookback_window: 126
  ranking_metric: risk_adjusted_momentum
  top_k: 5
  weighting: inverse_volatility
  max_weight: 0.30
  rebalance:
    mode: monthly
```

## 14. 深度学习与强化学习基准模型逻辑

### 14.1 PPO 连续权重策略 `PPO`

用途：

```text
使用 PPO 直接输出连续资产权重。
```

状态：

```text
state_t = {
  market_image_t,
  current_weights_t,
  availability_mask_t,
  risk_state_t optional
}
```

Actor：

```text
alpha_t = Actor(state_t)
w_t ~ Dirichlet(alpha_t) during training
w_t = alpha_t / sum(alpha_t) during deterministic evaluation
```

Critic：

```text
V_t = Critic(state_t)
```

执行逻辑：

```text
1. PPO actor 输出 candidate_weights。
2. ConstraintManager 修正权重。
3. 默认每个环境 step 都允许调仓。
4. 可通过 RebalanceScheduler 限制调仓频率。
5. 环境扣除交易成本并返回 reward。
6. 使用 clipped PPO objective 与 GAE 训练。
```

### 14.2 CNN-PPO `CNN-PPO`

用途：

```text
使用 CNN 编码 market_image，再由 PPO 输出连续权重。
```

模型结构：

```text
market_image -> CNN encoder -> latent z_t
z_t + current_weights + risk_state -> PPO actor
z_t + current_weights + risk_state -> PPO critic
```

执行逻辑与 PPO 相同，差异在于状态表征由 CNN 生成。

### 14.3 Bernoulli-Gated CNN-PPO

用途：

```text
使用 on-policy Bernoulli gate 判断是否调仓，用 CNN-PPO 输出候选权重。
```

模型输出：

```text
candidate_weights_t = PPOActor(z_t)
p_rebalance_t = sigmoid(GateHead(z_t, decision_weights_t, candidate_weights_t))
g_t ~ Bernoulli(p_rebalance_t)
```

执行逻辑：

```text
if g_t == 1:
    executed_weights = candidate_weights_t
else:
    executed_weights = pre_execution_drifted_weights_t
```

训练要求：

```text
gate 与 PPO actor 使用 on-policy log_prob。
必须记录 gate entropy、p_rebalance、rebalance frequency。
```

### 14.4 DQN-Only Template Selector

用途：

```text
使用 DQN 在离散组合模板中选择一个组合。
```

动作空间：

```text
0: equal_weight
1: minimum_variance
2: maximum_sharpe
3: risk_parity
4: inverse_volatility
5: defensive / money-market-heavy allocation
6: momentum_top_k
```

算法逻辑：

```text
1. 每个动作对应一个可计算的组合模板。
2. DQN 根据状态选择模板编号。
3. 模板函数生成 target_weights。
4. ConstraintManager 修正权重。
5. 环境执行并扣除交易成本。
6. DQN 使用 replay buffer、target network、Double DQN 训练。
```

限制：

```text
DQN 不直接输出连续权重。
连续权重由离散模板函数生成。
```

### 14.5 EIIE 风格模型

用途：

```text
使用共享子网络对每个资产独立评分，并生成归一化组合权重。
```

输入：

```text
per_asset_feature_tensor: [batch, n_assets, n_features, window_size]
previous_weights: [batch, n_assets]
```

模型逻辑：

```text
1. 对每个资产应用相同的 evaluator network。
2. 每个资产输出一个 score_i。
3. 可拼接 previous_weights 作为 portfolio-vector memory。
4. 对 score 做 softmax 得到权重。
5. 不可用资产 score 设置为 -inf。
6. 通过 ConstraintManager 修正。
```

执行要求：

```text
支持 daily rebalance 与 scheduled rebalance。
必须使用统一 CostModel。
必须输出日度权重、换手和成本。
```

## 15. 平台强化学习算法模块逻辑

### 15.1 A. Full DQN-Gated Multi-Task CNN-PPO

用途：

```text
使用 PPO 生成连续候选权重，使用 DQN 判断是否执行调仓，使用多任务辅助头增强市场表征。
```

执行顺序：

```text
1. 构造状态：
      decision_weights_t = PortfolioState.current_weights at decision_date close
      state_t = {
        market_image_t,
        decision_weights_t,
        optional_risk_state_t,
        availability_mask_t
      }

2. 编码状态：
      z_t = Encoder(state_t)

3. PPO 生成候选组合权重：
      alpha_t = PPOActor(z_t)
      candidate_w_t ~ Dirichlet(alpha_t) during training
      candidate_w_t = alpha_t / sum(alpha_t) during deterministic evaluation

4. 在 decision_date 估计交易规模和交易成本：
      estimated_turnover_t = 0.5 * sum_i |candidate_w_i,t - decision_weights_i,t|
      estimated_cost_t 只能使用 DecisionMarketState 中可见字段估算

5. DQN gate 输入：
      gate_input_t = concat(
        z_t,
        decision_weights_t,
        candidate_w_t,
        estimated_turnover_t,
        estimated_cost_t,
        optional_drawdown_t,
        optional_volatility_t,
        optional_regime_t
      )

6. DQN gate 决策：
      g_t = 0: hold current portfolio
      g_t = 1: rebalance to PPO candidate weights

7. 环境执行：
      if g_t == 1:
          executed_w_t = candidate_w_t
          apply transaction cost
      else:
          executed_w_t = pre_execution_drifted_weights_t
          transaction cost = 0

8. 环境更新 NAV、权重漂移、回撤和风险状态。

9. PPO 使用 on-policy rollout 更新。

10. DQN 使用 replay buffer 更新。

11. 辅助预测头只使用未来标签作为监督学习目标，不得进入 state_t。
```


PPO actor 更新样本选择：

```text
1. 若 executed_weights_t 来自 PPO candidate_weights_t，则该样本进入 PPO actor policy loss。
2. 若 executed_weights_t 来自 hold / drifted_weights，则该样本默认不进入 PPO actor policy loss。
3. gate=hold 样本仍可用于 critic、encoder 和 auxiliary heads 更新。
4. 若启用 counterfactual PPO update，必须显式计算 reward_if_rebalance 与 reward_if_hold，并记录 candidate_advantage。
```


Partial rebalance 下 PPO actor 更新规则：

```text
1. 若 rho_t = 0，则 candidate_weights_t 不进入 PPO actor policy loss。
2. 若 0 < rho_t < 1，则 candidate_weights_t 进入 PPO actor policy loss，默认 actor_loss_weight_t = rho_t。
3. 若 rho_t = 1，则 actor_loss_weight_t = 1。
4. 若 partial_rebalance.mode = continuous_beta，则 PPO joint log_prob = log_prob_dirichlet + log_prob_beta。
5. 若 partial_rebalance.mode = discrete_dqn，则 DQN 学习 rho 档位，PPO actor 只对 candidate_weights 的 Dirichlet log_prob 更新。
```

关键规则：

```text
DQN gate 必须看到 PPO candidate weights、estimated_turnover_t、estimated_cost_t。
DQN gate input 只能包含 decision_date 可见信息。
realized_turnover_t 由 PortfolioExecutionCore 在 execution_date 根据 pre_execution_drifted_weights 计算。
realized_cost_t 由 CostModel 在 execution_date 根据 ExecutionMarketState 计算。
训练 reward 和 NAV 只能使用 realized_cost_t。
环境 info、daily_rebalance.csv、daily_costs.csv 必须同时记录 estimated_cost_t 与 realized_cost_t。
```

训练损失：

```text
L_total = L_PPO_actor + c_v * L_PPO_critic - c_ent * entropy + L_DQN + lambda_aux * L_aux
```

DQN 训练：

```text
Double DQN
target Q-network
epsilon-greedy exploration
uniform replay buffer
Prioritized Experience Replay option
N-step DQN option
Dueling DQN option
NoisyNet option
Q-gap no-trade threshold option
cooldown / hysteresis option
```

```text
让同一个策略根据不同投资偏好输出不同组合和调仓行为。
```

```text
omega = [
  omega_return,
  omega_turnover,
  omega_drawdown,
  omega_cvar,
  omega_concentration
```

```text
omega_i >= 0
sum_i omega_i = 1
训练时可从 Dirichlet 分布采样 omega
评估时使用固定偏好集合
```

```text
reward_vector_t = [
```

```text
scalar_reward_t = dot(omega, reward_vector_t)
```

```text
4. PPO actor 基于 conditioned latent 输出 candidate_weights。
5. DQN gate 基于 conditioned gate_input 判断是否调仓。
```

```text
return_focused      = [0.70, 0.05, 0.10, 0.10, 0.05]
```

### 15.3 C. Uncertainty-Aware DQN-Gated PPO

用途：

```text
在调仓门控中显式考虑模型不确定性；模型越不确定，交易越保守。
```

支持不确定性估计方式：

```text
MC Dropout
shared encoder + multiple actor heads
shared encoder + multiple critic heads
bootstrap ensemble
full ensemble optional
```

候选权重集合：

```text
w_t^{candidate, k}, k = 1..K
```

权重均值：

```text
mean_w_t = mean_k(w_t^{candidate, k})
```

权重不确定性：

```text
weight_uncertainty_t = mean_i Var_k(w_{t,i}^{candidate, k})
```

价值不确定性：

```text
q_uncertainty_t = Var_k(Q_k(gate_input_t, action))
```

置信调整后的调仓收益：

```text
confidence_adjusted_benefit_t =
    expected_benefit_t
  - estimated_cost_t
  - kappa_q * sqrt(q_uncertainty_t)
  - kappa_w * weight_uncertainty_t
```

模型逻辑：

```text
1. Encoder 生成 z_t。
2. 多个 actor head 或 dropout forward 生成多个 candidate_weights。
3. 计算 mean_w_t、weight_uncertainty_t。
4. 多个 critic / Q head 估计 q_uncertainty_t。
5. DQN gate 输入加入 uncertainty features。
6. DQN 根据成本、收益改善和不确定性判断是否调仓。
7. 如果 confidence_adjusted_benefit_t 低于阈值，可强制 hold 或提高 hold 动作优势。
```

DQN gate 输入扩展：

```text
z_t
decision_weights_t
mean_candidate_weights_t
candidate_weight_variance_t
estimated_turnover_t
estimated_cost_t
q_uncertainty_t
weight_uncertainty_t
rolling_volatility_t
current_drawdown_t
```

配置：

```yaml
uncertainty_aware:
  method: multi_head
  n_heads: 5
  dropout_rate: 0.10
  kappa_q: 0.50
  kappa_w: 0.50
  force_hold_when_uncertainty_above: null
```

输出要求：

```text
uncertainty_mean
uncertainty_quantiles
uncertainty_vs_future_loss
uncertainty_vs_gate_action
rebalance_frequency_by_uncertainty_bucket
```

### 15.4 D. Distributional-CVaR DQN-Gated PPO

用途：

```text
使用分布式 critic 估计未来收益分布，并将下尾风险纳入调仓门控和策略评估。
```

分布式 critic 输出：

```text
Z_psi(s_t, w_t) = [z_tau1, z_tau2, ..., z_tauN]
```

CVaR 近似：

```text
CVaR_alpha(Z) = mean(z_tau_i for tau_i <= alpha)
```

风险调整价值：

```text
U_alpha(s_t, w_t) = mean(Z) - lambda_tail * tail_loss_alpha(Z)
```

其中：

```text
tail_loss_alpha(Z) = max(0, -CVaR_alpha(Z))
```

gate 决策信号：

```text
delta_U_t =
    U_alpha(s_t, candidate_w_t)
  - U_alpha(s_t, hold_w_t)
  - estimated_cost_t
```

DQN gate 可学习：

```text
g_t = 1 if expected long-term value of rebalance exceeds hold
```

也可启用规则阈值：

```text
g_t = 1 if delta_U_t > q_gap_threshold
```

模型逻辑：

```text
1. PPO actor 输出 candidate_weights。
2. Distributional critic 估计 candidate 与 hold 的收益分布。
3. 计算均值、低分位收益、CVaR、tail loss。
4. DQN gate 输入加入 distributional risk features。
5. 环境按 gate 结果执行。
6. critic 使用 quantile regression loss 训练。
7. PPO advantage 可基于 mean value 或 risk-adjusted value。
```

DQN gate 输入扩展：

```text
candidate_expected_value
candidate_CVaR_alpha
candidate_tail_loss
hold_expected_value
hold_CVaR_alpha
hold_tail_loss
delta_expected_value
delta_CVaR
delta_U
estimated_cost
```

配置：

```yaml
distributional_cvar:
  n_quantiles: 51
  cvar_alpha: 0.05
  lambda_tail: 0.50
  quantile_loss: huber
  use_risk_adjusted_advantage: true
```

输出要求：

```text
quantile_value_curve
cvar_curve
tail_loss_curve
cvar_gate_action_table
stress_period_metrics
```

### 15.5 E. Partial-Rebalance DQN / Beta-Gated PPO

用途：

```text
不只判断是否调仓，还需要决定调仓强度。
```

执行权重：

```text
executed_w_t =
    (1 - rho_t) * pre_execution_drifted_weights_t
  + rho_t * candidate_w_t
```

其中：

```text
rho_t = 0 表示不调仓
rho_t = 1 表示完全调仓到 candidate_weights
0 < rho_t < 1 表示部分调仓
```


Partial rebalance 执行后约束检查：

```text
Partial rebalance 执行后必须进行 executed_weights feasibility check。

若 executed_weights 违反 availability、long-only、simplex：
  必须立即修正。

若 executed_weights 违反 max_weight / HHI / turnover_limit：
  按 constraints.partial_rebalance_post_check_policy 处理：
    report_only
    project_executed
    force_full_rebalance

默认 partial_rebalance_post_check_policy = report_only。
默认不对 executed_weights 做强投影，避免实际成交强度偏离 rho_t。
```

支持三种实现模式。

#### 15.5.1 Discrete Partial DQN Gate

动作空间：

```text
0: hold, rho = 0.00
1: partial rebalance, rho = 0.25
2: partial rebalance, rho = 0.50
3: partial rebalance, rho = 0.75
4: full rebalance, rho = 1.00
```


DQN 输出维度规则：

```text
当 partial_rebalance.mode = discrete_dqn 时：
  dqn_gate.output_dim = len(partial_rebalance.discrete_rho_values)

当 partial_rebalance.mode = binary_gate 或 hybrid_dqn_beta 时：
  dqn_gate.output_dim = 2
```

算法逻辑：

```text
1. PPO actor 输出 candidate_weights。
2. DQN 输出离散动作。
3. 根据动作映射 rho。
4. 用 rho 计算 executed_weights。
5. 根据 executed_weights 计算 turnover 和 cost。
6. 使用 DQN replay buffer 更新离散强度选择。
```

#### 15.5.2 Continuous Beta Rebalance Intensity

Beta actor：

```text
rho_t ~ Beta(a_t, b_t)
```


Beta 强度数值规则：

```text
a_raw, b_raw -> softplus -> + beta_min_concentration
rho 必须位于 [0, 1]
训练 log_prob 优先使用未裁剪分布样本；仅允许出于数值安全做最终 clip
```

模型输出：

```text
candidate_weights_t ~ Dirichlet(alpha_t)
rho_t ~ Beta(a_t, b_t)
```

算法逻辑：

```text
1. PPO actor 同时输出 Dirichlet 参数 alpha_t 和 Beta 参数 a_t, b_t。
2. 训练时采样 candidate_weights 与 rho。
3. 评估时使用分布均值。
4. 环境执行部分调仓。
5. PPO log_prob = log_prob_dirichlet + log_prob_beta。
```

#### 15.5.3 Hybrid DQN + Beta Intensity

逻辑：

```text
1. DQN 先判断是否允许交易。
2. 如果 DQN 输出 hold，则 rho = 0。
3. 如果 DQN 输出 rebalance，则 Beta actor 输出 rho。
4. executed_weights = pre_execution_drifted_weights + rho * (candidate - pre_execution_drifted_weights)。
```

配置：

```yaml
partial_rebalance:
  mode: discrete_dqn
  discrete_rho_values: [0.0, 0.25, 0.50, 0.75, 1.0]
  beta_min_concentration: 1.0e-3
  log_prob_includes_beta: true
```

输出要求：

```text

transaction_cost_distribution
```


### 16.2 PPO Actor


要求：


Dirichlet mask 语义：

```text
1. 不允许先从全资产 Dirichlet 采样，再对 unavailable asset 硬裁剪并重归一化。
2. 必须在分布构造阶段只对 available assets 建立 Dirichlet。
3. log_prob 只在 available asset 子空间计算。
4. 采样后的 available weights 再散射回 n_assets 维向量，不可用资产权重为 0。
5. deterministic evaluation 使用 available alpha 的均值 alpha / sum(alpha)。
6. 若 available assets 少于 1 个，按 ConstraintManager 不可行兜底处理。
```


数值稳定规则：

```text
alpha_raw -> softplus -> alpha + min_alpha
alpha must be finite
若任一 alpha 为 NaN 或 Inf，必须 fail fast
若 available asset count == 1，deterministic weight = 1.0，policy log_prob 记为 0 或跳过 actor loss
```
