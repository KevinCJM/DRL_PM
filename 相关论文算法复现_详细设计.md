# 相关论文算法复现_详细设计

## 1. 修改摘要 (Revision Summary)

- 实现目标：
  - 在平台原生回测、训练、HPO、论文汇总链路中新增两个论文相关算法复现能力：
    - `ppo_dqn_hierarchical_reimplementation` [PPO-DQN 层级动作选择复现模型]：由 PPO 产生候选组合权重，由 DQN 选择五类平台适配执行动作，最终通过平台约束和回测引擎落地。
    - `hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族编排标识]：只作为编排 alias，展开为五个可训练 child run，不生成可排名主表行。
  - 正式可排名实验只包含六个 trainable entries：
    - `ppo_dqn_hierarchical_reimplementation` [PPO-DQN 层级复现可训练条目]
    - `hybrid_dqn_optimizer_equal_weight` [DQN 候选池 + 等权优化器可训练条目]
    - `hybrid_dqn_optimizer_markowitz_mean_variance` [DQN 候选池 + Markowitz 均值方差优化器可训练条目]
    - `hybrid_dqn_optimizer_minimum_variance` [DQN 候选池 + 最小方差优化器可训练条目]
    - `hybrid_dqn_optimizer_sharpe_maximization` [DQN 候选池 + Sharpe 最大化优化器可训练条目]
    - `hybrid_dqn_optimizer_risk_parity` [DQN 候选池 + 风险平价优化器可训练条目]
  - HPO、最终测试、诊断 sidecar、论文主表统一使用六个 trainable entries；`hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族编排标识] 只允许作为配置编排 alias 或非排名诊断标识，不写入可排名输出契约；`baseline_family` [baseline 算法族] 固定为 `native_rl_reimplementation`。

## 2. 业务逻辑与代码映射 (Logic Mapping)

### 2.1 Baseline 注册与 alias 展开

- 【位置】：`src/experiments/registry.py` -> `DEEP_BASELINE_CLASSES` [平台原生 deep baseline 类注册表]。
- 【执行流描述】：
  - 前置条件：
    - `baselines.native_rl.enabled_models` [配置中启用的平台原生模型列表] 或 `baselines.deep` [配置中启用的 deep baseline 列表] 含本次新增模型 id。
    - `src/baselines/__init__.py` [baseline 类导出入口] 已导出六个 trainable strategy class。
  - 核心步骤：
    1. 在 `DEEP_BASELINE_CLASSES` [平台原生 deep baseline 类注册表] 中注册六个 trainable entries。
    2. 新增 `HYBRID_DQN_OPTIMIZER_ALIAS` [Hybrid DQN Optimizer 算法族 alias]，取值固定为 `hybrid_dqn_optimizer_reimplementation`。
    3. 新增 `HYBRID_DQN_OPTIMIZER_CHILD_MODEL_NAMES` [Hybrid DQN Optimizer 五个可训练 child 模型列表]，顺序固定为：
       - `hybrid_dqn_optimizer_equal_weight`
       - `hybrid_dqn_optimizer_markowitz_mean_variance`
       - `hybrid_dqn_optimizer_minimum_variance`
       - `hybrid_dqn_optimizer_sharpe_maximization`
       - `hybrid_dqn_optimizer_risk_parity`
    4. 新增 `_expand_baseline_aliases(model_names)` [将配置中的算法族 alias 展开为可训练模型 id 的函数]。
    5. `_baseline_factories(config)` [根据实验配置生成 baseline factory 的函数] 在读取 `baselines.deep`、`baselines.native_rl.enabled_models`、`baselines.native` 后，对模型列表调用 `_expand_baseline_aliases(model_names)` [alias 展开函数]。
    6. `_baseline_factories(config)` [baseline factory 生成函数] 对展开后的每个 model id 查找 `DEEP_BASELINE_CLASSES` [平台原生 deep baseline 类注册表]，生成独立 factory。
    7. 当配置中直接出现五个 child model id 时，直接生成五个 factory，不重复展开。
    8. 当配置中出现 `hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族 alias] 时，只生成五个 child factory，不生成 alias 自身 factory。
  - 分支判断：
    - `if model_name == HYBRID_DQN_OPTIMIZER_ALIAS` [当前配置项是算法族 alias]：
      - 输出五个 child model names。
      - 不输出 alias 本身。
    - `elif model_name in HYBRID_DQN_OPTIMIZER_CHILD_MODEL_NAMES` [当前配置项是可训练 child id]：
      - 原样输出该 child id。
    - `else` [当前配置项是其他已有 baseline id]：
      - 保持原有注册逻辑。
    - `if expanded_model_name not in DEEP_BASELINE_CLASSES` [展开后的模型 id 未注册]：
      - 保持现有未知模型报错路径，禁止吞错。
- 【关键变量说明】：
  - `DEEP_BASELINE_CLASSES` [平台原生 deep baseline 类注册表]：
    - 类型/范围：`dict[str, type]`。
    - 取值说明：key 为平台内部 model id；value 为 strategy class。
    - 备注：必须包含六个 trainable entries，不包含 `hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族 alias] 的可训练类。
  - `HYBRID_DQN_OPTIMIZER_ALIAS` [Hybrid DQN Optimizer 算法族 alias]：
    - 类型/范围：`str`，固定值 `hybrid_dqn_optimizer_reimplementation`。
    - 取值说明：只用于配置编排和元数据族标识。
    - 备注：不得作为 rankable `paper_model_id` [论文主表可排名模型标识]。
  - `HYBRID_DQN_OPTIMIZER_CHILD_MODEL_NAMES` [Hybrid DQN Optimizer 五个 child 模型列表]：
    - 类型/范围：`tuple[str, ...]`。
    - 取值说明：五个可训练、可 HPO、可排名 child id。
    - 备注：顺序作为默认展示和展开顺序，不作为排名顺序。
  - `model_name` [平台运行时模型名]：
    - 类型/范围：`str`。
    - 取值说明：在六个 trainable entries 中表示真实训练单元；在 alias 中只表示编排请求。
    - 备注：诊断 sidecar 中允许保留 `model_name` [平台运行时模型名]，但必须同时写 `paper_model_id` [论文可排名模型标识] 消除歧义。

### 2.2 HPO native-only 过滤与 final sidecar 聚合

- 【位置】：`src/experiments/run_experiment.py` -> `NATIVE_HPO_MODEL_NAMES` [允许走平台原生 HPO 的模型名集合]。
- 【执行流描述】：
  - 前置条件：
    - HPO 配置启用 `hpo.native_only` [只保留平台原生可训练模型的 HPO 过滤开关]。
    - 配置中的 trainable model list 来自 `hpo.trainable_models` [显式 HPO 模型列表] 或默认 baseline 列表。
    - 正式 main comparable table 使用 `random_seeds` [正式随机种子集合] 逐 seed 独立 HPO，不允许使用单次 HPO best config 后跨 seed retrain 的口径。
    - 编排配置设置 `long_running=true` [长任务编排标记]，并启用 checkpoint、resume、progress logging 或定时检查。
  - 核心步骤：
    1. 在 `NATIVE_HPO_MODEL_NAMES` [允许走平台原生 HPO 的模型名集合] 中加入六个 trainable entries。
    2. 不把 `hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族 alias] 加入 `NATIVE_HPO_MODEL_NAMES` [允许走平台原生 HPO 的模型名集合]。
    3. `_hpo_trainable_models(config)` [解析 HPO 可训练模型列表的函数] 在过滤前调用 `_expand_baseline_aliases(model_names)` [alias 展开函数]。
    4. `_filter_hpo_proxy_models(model_names)` [过滤非 HPO 原生模型的函数] 保持现有非 native 过滤口径，仅接收已展开后的六个 trainable ids。
    5. `_is_native_hpo_trainable_model(model_name)` [判断模型是否允许走 native HPO 的函数] 对六个 trainable entries 返回 `True`。
    6. `_is_native_hpo_trainable_model(model_name)` [判断模型是否允许走 native HPO 的函数] 对 `hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族 alias] 返回 `False`。
    7. 对每个 `seed` [随机种子] 独立创建 HPO search space、trial 记录、validation selection 与 best checkpoint，不复用其他 seed 的 best config。
    8. 对每个 `seed` [随机种子] 内的六个 trainable entries 分配相同 `hpo_trials` [HPO trial 数]、`env_steps` [环境步数]、`gradient_updates` [梯度更新次数]、`validation_metric` [验证指标] 与 `selection_rule` [选择规则]。
    9. 每个 `seed` [随机种子] 的 final test 只加载该 seed 自己 validation 选出的 `checkpoint_best_path` [最佳 checkpoint 路径]，并写入 `evaluated_checkpoint_path` [实际评估 checkpoint 路径]。
    10. HPO equal-budget final artifact 聚合时，将每个 child run 的 `baseline_daily_diagnostics` [baseline 每日诊断 sidecar 数据] 聚合为 `hpo_model_final_daily_diagnostics` [HPO 每模型最终每日诊断聚合表]。
    11. 聚合输出路径固定为 `metrics/hpo_model_final_daily_diagnostics.csv` [HPO per-model final 诊断 sidecar 聚合文件]。
    12. 正式表跨 seed 聚合均值、标准差、最小值、最大值和中位数。
  - 分支判断：
    - `if hpo.trainable_models is explicit` [HPO 显式指定训练模型]：
      - 读取显式列表。
      - 对 alias 展开。
      - 再执行 native-only 过滤。
    - `else` [HPO 未显式指定训练模型]：
      - 从默认主模型和 baseline 配置生成候选。
      - 对 alias 展开。
      - 再执行 native-only 过滤。
    - `if model_name == HYBRID_DQN_OPTIMIZER_ALIAS` [HPO 候选仍是算法族 alias]：
      - 该分支在展开后不应出现。
      - 若出现，视为配置解析错误，不作为 budget unit。
    - `if child_payload.status != completed and run_mode not in {"smoke", "diagnostic"}` [正式/HPO/rankable 的某个必需 child 失败]：
      - 正式/HPO/rankable run 顶层状态为 `failed` [运行失败]。
      - CLI 返回失败。
    - `if run_mode in {"smoke", "diagnostic"}` [显式 smoke 或 diagnostic 运行]：
      - 五个 Hybrid DQN optimizer child 的 `status` [运行状态] 全部为 `completed` [完成] 时，算法族 smoke 顶层状态写 `completed` [完成]。
      - 任一 child `failed` [失败] 或 `skipped` [跳过] 时，显式 smoke/diagnostic run 顶层 `status` [运行状态] 可以写 `completed` [完成]，但不得写 `partial_diagnostic` [部分诊断]。
      - 必须写入 `diagnostic_status=partial_diagnostic` [部分诊断状态]、`rankable_in_unified_table=false` [禁止进入统一主表排名]、`failed_child_model_id` [失败 child 模型标识] 与 `reason` [失败原因]。
      - 该 partial diagnostic 结果只能进入 diagnostic table 或 appendix，不进入 `paper_main_comparison.csv` [论文主对比表]、`paper_seed_summary.csv` [论文 seed 汇总表] 或正式配对统计。
    - `if protocol == "shared_dqn_multi_optimizer_final_test"` [共享 DQN 后多优化器 final test 协议]：
      - 只允许作为 `diagnostic_status=diagnostic_shared_dqn` [共享 DQN 诊断方案]。
      - 写入 `rankable_in_unified_table=false` [禁止进入统一主表排名]。
      - 不进入 `paper_main_comparison.csv` [论文主比较表]、`paper_seed_summary.csv` [论文 seed 汇总表] 或正式配对统计。
- 【关键变量说明】：
  - `NATIVE_HPO_MODEL_NAMES` [允许走平台原生 HPO 的模型名集合]：
    - 类型/范围：`set[str]`。
    - 取值说明：包含既有 native baseline 和本次六个 trainable entries。
    - 备注：不包含算法族 alias。
  - `hpo.native_only` [只保留平台原生可训练模型的 HPO 过滤开关]：
    - 类型/范围：`bool`。
    - 取值说明：`true` 时只保留 `_is_native_hpo_trainable_model(model_name)` [native HPO 可训练判断] 为真的模型。
    - 备注：过滤对象是 trainable entry，不是算法族。
  - `hpo_model_final_daily_diagnostics` [HPO 每模型最终每日诊断聚合表]：
    - 类型/范围：`pandas.DataFrame`。
    - 取值说明：由各模型 final test 的 `baseline_daily_diagnostics` [baseline 每日诊断 sidecar 数据] 拼接得到。
    - 备注：必须包含 `paper_model_id` [论文可排名模型标识]。
  - `seed` [随机种子]：
    - 类型/范围：`int`。
    - 取值说明：正式 HPO 的隔离单位。
    - 备注：每个 seed 独立 trial、独立 best checkpoint、独立 final test。
  - `long_running` [长任务编排标记]：
    - 类型/范围：`bool`，正式 per-seed HPO 固定为 `true`。
    - 取值说明：指示编排层启用 checkpoint、resume、progress logging 或定时检查。
    - 备注：smoke run 可使用短预算，但不得替代正式 per-seed HPO。

### 2.3 Validation metric alias 与训练失败判定

- 【位置】：`src/experiments/pipeline.py` -> `objective_metric(result, metric)` [从实验结果中读取 HPO 目标指标的函数]。
- 【执行流描述】：
  - 前置条件：
    - HPO 或训练验证阶段请求 `validation_penalized_sharpe` [论文需求别名] 或 `validation_sharpe_minus_drawdown_turnover_penalty` [平台 canonical validation 指标 key]。
    - Validation split 已通过 `BacktestEngine` [平台回测引擎] 生成 net daily returns。
  - 核心步骤：
    1. 新增 `VALIDATION_METRIC_ALIASES` [validation 指标 alias 映射表]。
    2. `VALIDATION_METRIC_ALIASES["validation_penalized_sharpe"]` [论文需求别名到平台 canonical key 映射] 固定指向 `validation_sharpe_minus_drawdown_turnover_penalty`。
    3. `objective_metric(result, metric)` [HPO 目标指标读取函数] 首先将 `metric` [请求指标名] 规范化为 canonical key。
    4. canonical key 为 `validation_sharpe_minus_drawdown_turnover_penalty` 时，沿用现有计算语义：
       - `net_return_sharpe` [validation net daily returns Sharpe]
       - 减去 `max_drawdown(nav)` [validation 净值最大回撤]
       - 减去 `average_turnover` [validation 平均换手率]
    5. 训练阶段 best checkpoint selection 使用同一 canonical key。
    6. 若计算结果不是 finite number，则训练或 HPO trial 返回 `failed_no_finite_validation_metric` [无有限 validation 指标导致失败]。
    7. 禁止使用训练 reward 替代 validation metric。
    8. 禁止新增 baseline 专属 validation 指标。
  - 分支判断：
    - `if metric == "validation_penalized_sharpe"` [请求论文需求别名]：
      - 转换为 `validation_sharpe_minus_drawdown_turnover_penalty` [平台 canonical validation 指标 key]。
    - `elif metric == "validation_sharpe_minus_drawdown_turnover_penalty"` [请求平台 canonical key]：
      - 原样使用。
    - `elif metric == "net_sharpe"` [已有平台兼容指标]：
      - 保持现有处理。
    - `else` [未知指标 key]：
      - 保持现有未知指标报错。
    - `if not np.isfinite(metric_value)` [validation 指标不是有限数]：
      - strategy `fit()` [模型训练入口] 返回失败状态 `failed_no_finite_validation_metric`。
- 【关键变量说明】：
  - `validation_penalized_sharpe` [论文需求指标别名]：
    - 类型/范围：`str`。
    - 取值说明：只在需求、论文语义或配置兼容层出现。
    - 备注：代码内部 canonical key 不使用该值作为最终指标名。
  - `validation_sharpe_minus_drawdown_turnover_penalty` [平台 canonical validation 指标 key]：
    - 类型/范围：`str`。
    - 取值说明：正式配置和代码统一使用的指标 key。
    - 备注：计算输入必须来自 validation split 的 `BacktestEngine` [平台回测引擎] net daily returns。
  - `metric_value` [validation 指标数值]：
    - 类型/范围：`float`，必须为 finite。
    - 取值说明：数值越大表示验证表现越好。
    - 备注：非 finite 时不得参与 best checkpoint selection。

### 2.4 PPO-DQN 层级复现模型

- 【位置】：`src/baselines/ppo_dqn_hierarchical_reimplementation.py` -> `PPODQNHierarchicalReimplementationStrategy` [PPO-DQN 层级复现策略类]。
- 【执行流描述】：
  - 前置条件：
    - `fit(train_data, validation_data)` [训练入口] 收到非空 train split 和 validation split。
    - `PortfolioRebalanceEnv` [组合调仓训练环境] 可提供 `current_weights` [当前组合权重]、`availability_mask` [可用资产掩码]、市场特征与账户状态。
    - `observation` [PPO-DQN 决策输入] 只包含 decision time 可见字段，不包含 execution date 后数据、next valuation 后收益、`realized_cost` [已实现成本]、`realized_turnover` [已实现换手] 或其他 execution-only 字段。
    - PPO 子模块能输出 `candidate_weights` [PPO 候选组合权重] 和 `candidate_log_prob` [PPO 候选动作对数概率]。
    - DQN 子模块能对五个 `hierarchy_action` [层级执行动作] 输出 Q 值。
  - 核心步骤：
    1. `fit()` [训练入口] 构造 train env 和 validation env。
    2. 初始化 PPO actor/critic 网络、DQN hierarchy action 网络、优化器、replay buffer。
    3. 每个 train step 先调用平台现有 decision-visibility 校验，确认 `observation` [PPO-DQN 决策输入] 不含 execution-only 字段。
    4. 每个 train step 读取 observation：
       - `candidate_weights_raw` [PPO 原始候选权重]
       - `current_weights_raw` [环境当前权重]
       - `available_mask_at_decision` [决策时可用资产掩码]
    5. 调用 `_mask_normalize_weights(candidate_weights_raw, available_mask_at_decision)` [按可用资产掩码归一化权重函数] 得到 `candidate_weights` [PPO 可执行候选权重]。
    6. 若 `candidate_weights` [PPO 可执行候选权重] 不可归一化、含负值或含非 finite 值，则记录 `candidate_weights_valid=false` [候选权重无效标记] 与 `candidate_invalid_reason` [候选权重无效原因]；除非后续 `hierarchy_action==4` [可用资产等权 fallback]，否则本 transition 失败为 `failed_no_valid_action` [无有效动作失败]。
    7. 调用 `_mask_normalize_weights(current_weights_raw, available_mask_at_decision)` [按可用资产掩码归一化权重函数] 得到 `current_weights` [可执行当前权重]。
    8. DQN 选择 `hierarchy_action` [层级执行动作]。
    9. 调用 `_resolve_hierarchy_action(hierarchy_action, candidate_weights, current_weights, available_mask_at_decision)` [将层级动作解析为目标权重与归因信息的函数]。
    10. 将解析后的 `target_weights` [目标组合权重] 放入 `PortfolioAction.target_weights` [组合目标权重字段]。
    11. `PortfolioAction` [组合动作对象] 进入 `ConstraintManager` [组合约束管理器] 和 `BacktestEngine` [平台回测引擎]。
    12. 根据真实 env transition 更新 DQN。
    13. 根据 `ppo_actor_update_mask` [PPO actor 是否更新掩码] 与 `ppo_attribution_weight` [PPO 收益归因权重] 更新 PPO actor。
    14. PPO critic 始终使用真实 transition 训练。
    15. 每个 validation checkpoint 使用 `validation_sharpe_minus_drawdown_turnover_penalty` [平台 canonical validation 指标 key] 选 best checkpoint。
    16. `compute_target_weights()` [推理阶段目标权重计算入口] 加载 best checkpoint，并执行同一 `_resolve_hierarchy_action()` [层级动作解析函数]。
    17. `training_result` [训练结果]、`baseline_training_summary.csv` [baseline 训练摘要表]、`baseline_comparison.csv` [baseline 对比表] 写入 PPO-DQN 最低元数据：`model_name=ppo_dqn_hierarchical_reimplementation`、`baseline_family=native_rl_reimplementation`、`training_algorithm=ppo_dqn_hierarchical_reimplementation`、`clean_room_reimplementation=true`、`algorithm_fidelity=platform_adapted`、`dqn_role=high_level_action_selector`、`platform_native_rl_training=true`、`proxy_training=false`、`rankable_in_unified_table=true`。
  - 分支判断：
    - `if hierarchy_action == 0` [使用 PPO candidate 权重]：
      - `hierarchy_action_name="use_ppo_candidate"` [层级动作名称]。
      - `target_weights = candidate_weights` [目标权重等于 PPO 候选权重]。
      - `ppo_actor_update_mask=1` [PPO actor 参与更新]。
      - `ppo_attribution_weight=1.0` [PPO 收益归因权重为 1]。
      - `platform_adapted_surrogate=false` [非 surrogate blend]。
    - `elif hierarchy_action == 1` [25% PPO blend]：
      - `rho=0.25` [PPO 候选权重混合比例]。
      - `target_weights=(1-rho)*current_weights + rho*candidate_weights` [blend 后目标权重]。
      - `ppo_actor_update_mask=1` [PPO actor 参与更新]。
      - `ppo_attribution_weight=0.25` [PPO 收益归因权重]。
      - PPO actor loss 使用 `candidate_log_prob * rho` [PPO candidate 对数概率乘归因权重]。
      - `platform_adapted_surrogate=true` [平台适配 surrogate 标记]。
    - `elif hierarchy_action == 2` [50% PPO blend]：
      - `rho=0.50`。
      - 其余规则与 25% blend 相同。
    - `elif hierarchy_action == 3` [75% PPO blend]：
      - `rho=0.75`。
      - 其余规则与 25% blend 相同。
    - `elif hierarchy_action == 4` [可用资产等权 fallback]：
      - `target_weights = equal_weight(available_assets)` [可用资产等权]。
      - `ppo_actor_update_mask=0` [PPO actor 不更新]。
      - `ppo_attribution_weight=0.0` [PPO 收益归因权重为 0]。
      - PPO actor loss 跳过或乘以 0。
      - PPO critic 仍训练。
      - DQN 仍按真实 transition 更新。
      - `platform_adapted_surrogate=false`。
      - 若 `candidate_weights_valid=false` [PPO 候选权重无效标记]，必须写入 `fallback_reason="invalid_candidate_weights_equal_weight"` [候选权重无效后等权 fallback 原因]，且仍保持 `ppo_actor_update_mask=0` [PPO actor 不更新] 与 `ppo_attribution_weight=0.0` [PPO 收益归因权重为 0]。
    - `if available_asset_count == 0` [无可用资产]：
      - 返回失败状态 `failed_no_valid_action` [无有效动作失败]。
    - `if candidate_weights_valid == false and hierarchy_action in {0, 1, 2, 3}` [候选权重无效但高层动作需要 PPO candidate]：
      - 返回失败状态 `failed_no_valid_action` [无有效动作失败]。
      - 写入 `reason="invalid_candidate_weights"` [候选权重无效原因]。
      - 不允许静默切到 equal available fallback。
      - 不更新 PPO actor。
    - `if target_weights` [目标权重] 经过平台约束后非 finite 或不可行：
      - 当前 step 标记 action failure。
      - 训练或 trial 返回 `failed_no_valid_action` [无有效动作失败]。
      - 写入 `reason="constraint_projection_failed"` [约束投影失败原因]。
      - 不进入 fallback 路径，禁止将非 finite 权重传入回测收益计算。
    - `if observation contains execution-only field` [决策输入含执行期字段]：
      - 训练或 trial 失败，禁止进入 validation、checkpoint selection 或 final test。
    - `if dqn_role != "high_level_action_selector"` [DQN 角色不是高层动作选择器]：
      - 该 run 不满足首版正式验收。
      - 写入 `status=deferred_variant` [延期变体] 或 `status=needs_paper_confirmation` [需要论文细节确认]。
      - 写入 `rankable_in_unified_table=false` [禁止进入统一主表排名]。
      - 禁止将该 run 误标为本项目 `rebalance_gate` [主模型调仓门控]，也不得进入首版 main comparable table。
- 【关键变量说明】：
  - `hierarchy_action` [DQN 层级执行动作编号]：
    - 类型/范围：`int`，取值 `0` 到 `4`。
    - 取值说明：
      - `0`：`use_ppo_candidate` [使用 PPO 候选权重]
      - `1`：`blend_current_ppo_25` [当前权重与 PPO 候选权重按 25% PPO 混合]
      - `2`：`blend_current_ppo_50` [当前权重与 PPO 候选权重按 50% PPO 混合]
      - `3`：`blend_current_ppo_75` [当前权重与 PPO 候选权重按 75% PPO 混合]
      - `4`：`fallback_equal_weight` [可用资产等权 fallback]
    - 备注：首版只允许这五个动作。
  - `candidate_weights` [PPO 候选组合权重]：
    - 类型/范围：`np.ndarray[float]`，长度等于资产数，非负，按可用资产归一化后和为 1。
    - 取值说明：PPO actor 提出的候选组合。
    - 备注：不可用资产权重强制为 0。
  - `candidate_weights_valid` [PPO 候选权重有效标记]：
    - 类型/范围：`bool`。
    - 取值说明：`candidate_weights_raw` [PPO 原始候选权重] 经 mask 后是否非负、finite 且可归一化。
    - 备注：若为 `false`，只有 `hierarchy_action=4` [可用资产等权 fallback] 可继续执行，且必须 `ppo_actor_update_mask=0` [PPO actor 不更新]；其他 hierarchy action 直接失败为 `failed_no_valid_action` [无有效动作失败]。
  - `current_weights` [当前可执行组合权重]：
    - 类型/范围：`np.ndarray[float]`，长度等于资产数，非负，按可用资产归一化后和为 1。
    - 取值说明：执行 blend 前的组合状态。
    - 备注：不可用资产权重强制为 0。
  - `rho` [PPO 候选权重混合比例]：
    - 类型/范围：`float`，只允许 `0.25`、`0.50`、`0.75`。
    - 取值说明：blend 动作中 PPO 候选权重的贡献比例。
    - 备注：同值写入 `ppo_attribution_weight` [PPO 收益归因权重]。
  - `ppo_actor_update_mask` [PPO actor 更新掩码]：
    - 类型/范围：`int`，取值 `0` 或 `1`。
    - 取值说明：`1` 表示 PPO actor 更新；`0` 表示本次 transition 不更新 PPO actor。
    - 备注：fallback equal weight 时固定为 `0`。
  - `ppo_attribution_weight` [PPO 收益归因权重]：
    - 类型/范围：`float`，取值 `0.0`、`0.25`、`0.50`、`0.75`、`1.0`。
    - 取值说明：actor loss 对 PPO candidate log_prob 的乘数。
    - 备注：用于避免收益归因不清。
  - `platform_adapted_surrogate` [平台适配 surrogate 标记]：
    - 类型/范围：`bool`。
    - 取值说明：daily diagnostics 中 blend 动作固定为 `true`，非 blend 动作为 `false`；training summary、training_result 与 baseline comparison 中为 run-level OR 汇总，即本 run 是否至少出现一次 blend actor-loss attribution。
    - 备注：必须进入 daily diagnostics、training summary、baseline comparison；summary 级字段不得解释为每个 step 都使用 surrogate。

### 2.5 Hybrid DQN Optimizer 五个 child 模型

- 【位置】：`src/baselines/hybrid_dqn_optimizer_reimplementation.py` -> `HybridDQNOptimizerReimplementationStrategy` [Hybrid DQN Optimizer 复现策略基类]。
- 【执行流描述】：
  - 前置条件：
    - 当前 child model id 为五个 `hybrid_dqn_optimizer_*` [Hybrid DQN Optimizer 可训练 child 模型] 之一。
    - `PortfolioRebalanceEnv` [组合调仓训练环境] 能提供资产级 observation 和 `availability_mask` [可用资产掩码]。
    - `DecisionMarketState.log_return_window` [决策时可见收益历史窗口] 已按 decision date 截断，供传统优化器估计收益、协方差或风险贡献。
    - `observation` [Hybrid DQN 决策输入]、`asset_action_q_values` [每资产三动作 Q 值] 生成链路和 `DecisionMarketState.log_return_window` [决策时可见收益历史窗口] 均不包含 execution date 后数据、next valuation 后收益、`realized_cost` [已实现成本]、`realized_turnover` [已实现换手] 或其他 execution-only 字段。
    - 每个 child run 单独训练一个 DQN 信号模型，单独执行 validation selection 和 final test。
  - 核心步骤：
    1. `fit()` [训练入口] 构造 train env 和 validation env。
    2. 初始化资产级 DQN 网络，输出 shape 为 `[asset_count, 3]` 的 `asset_action_q_values` [每资产三动作 Q 值]。
    3. 每个 step 先调用平台现有 decision-visibility 校验，确认 DQN signal 输入和 optimizer window 均只含 decision-visible 字段。
    4. 每个 step 对每个资产计算三类动作 Q：
       - `0 exclude` [排除资产]
       - `1 neutral` [中性保留资产]
       - `2 include` [纳入候选池资产]
    5. 对 `available_mask_at_decision=false` [决策时不可用资产] 的资产强制动作 `exclude`。
    6. 统计原始 include 资产数为 `selected_asset_count` [DQN 原始 include 数量]。
    7. 将原始 include 资产放入 `candidate_asset_indices` [优化器候选资产索引集合]。
    8. 若 `selected_asset_count < min_selected_assets` [原始 include 数少于最小候选资产数] 且 `available_asset_count >= min_selected_assets` [可用资产数满足最小候选资产数]，从所有非 include 且可用资产中按 `q_include - max(q_neutral, q_exclude)` [补足排序分数] 降序补足到 2。
    9. 补足后统计 `optimizer_asset_count` [进入优化器的最终资产数量]。
    10. 根据 child model 的 `optimizer_name` [组合优化器名称] 和 `DecisionMarketState.log_return_window` [决策时可见收益历史窗口] 调用 `_compute_optimizer_weights()` [候选池组合权重计算函数]。
    11. 将候选池权重映射回全资产维度，生成 `PortfolioAction.target_weights` [组合目标权重字段]。
    12. `PortfolioAction` [组合动作对象] 进入 `ConstraintManager` [组合约束管理器] 和 `BacktestEngine` [平台回测引擎]。
    13. DQN 使用组合级真实环境 reward 更新，`portfolio_level_reward_shared=true` [组合级 reward 共享]，`counterfactual_asset_reward=false` [不使用资产级反事实 reward]。
    14. validation split 使用 `validation_sharpe_minus_drawdown_turnover_penalty` [平台 canonical validation 指标 key] 选择 best checkpoint。
    15. `training_result` [训练结果]、`baseline_training_summary.csv` [baseline 训练摘要表]、`baseline_comparison.csv` [baseline 对比表] 写入 Hybrid DQN 最低元数据：`model_name=hybrid_dqn_optimizer_reimplementation`、`paper_model_id` [论文模型标识]、`child_model_name=paper_model_id`、`baseline_family=native_rl_reimplementation`、`training_algorithm=factorized_dqn_signal_plus_portfolio_optimizer`、`clean_room_reimplementation=true`、`factorized_q=true`、`portfolio_level_reward_shared=true`、`counterfactual_asset_reward=false`、`platform_adapted_approximation=true`、`platform_native_rl_training=true`、`proxy_training=false`、`rankable_in_unified_table=true`。
  - 分支判断：
    - `if available_asset_count == 0` [无可用资产]：
      - 返回失败状态 `failed_no_valid_action` [无有效动作失败]。
      - 不调用优化器。
    - `elif available_asset_count == 1` [仅一个可用资产]：
      - `target_weights` [目标权重] 对该资产赋 100%。
      - `fallback_reason="single_available_asset"` [仅单可用资产 fallback 原因]。
      - 标记 fallback。
      - 允许继续训练和回测。
    - `elif selected_asset_count >= 2` [原始 include 数量满足候选池下限]：
      - 候选池只包含原始 include 资产。
      - `candidate_fill_source="none"` [无补足来源]。
    - `elif selected_asset_count < 2 and available_asset_count >= 2` [原始 include 不足但可用资产足够]：
      - 从所有非 include 且可用资产补足。
      - 补足排序分数为 `q_include - max(q_neutral, q_exclude)` [include 相对非 include 优势分数]。
      - 允许补足来源为原始 neutral 或原始 exclude。
      - 补足到 `optimizer_asset_count=2`。
    - `if optimizer_name == "equal_weight"` [等权优化器 child]：
      - 只对 DQN 候选池等权。
      - 不读取协方差参数。
    - `elif optimizer_name == "markowitz_mean_variance"` [Markowitz 均值方差 child]：
      - 使用 `lookback_window=252` [收益估计窗口]。
      - 使用 `min_observations=60` [最小有效观测数]。
      - 使用 `covariance_shrinkage=0.1` [协方差收缩系数]。
      - 使用 `risk_free_rate=0.0` [无风险利率]。
      - 使用 `lambda_risk=1.0` [风险惩罚系数]。
      - 使用 `optimizer_maxiter=200` [优化器最大迭代次数]。
    - `elif optimizer_name == "minimum_variance"` [最小方差 child]：
      - 使用 `lookback_window=252`、`min_observations=60`、`covariance_shrinkage=0.1`、`optimizer_maxiter=200`。
    - `elif optimizer_name == "sharpe_maximization"` [Sharpe 最大化 child]：
      - 使用 `lookback_window=252`、`min_observations=60`、`covariance_shrinkage=0.1`、`risk_free_rate=0.0`、`optimizer_maxiter=200`。
    - `elif optimizer_name == "risk_parity"` [风险平价 child]：
      - 使用 `lookback_window=252`、`min_observations=60`、`covariance_shrinkage=0.1`、`optimizer_maxiter=300`。
    - `if optimizer_result.success == false` [传统优化器失败]：
      - 最终 fallback 为候选池等权。
      - 若候选池为空，则 fallback 为全部可用资产等权。
      - 写入 `fallback_reason` [fallback 原因] 和 `optimizer_status` [优化器状态]。
    - `if DecisionMarketState.log_return_window is empty` [决策可见收益窗口为空]：
      - `optimizer_status="failed_insufficient_history"` [历史不足导致优化器失败]。
      - `fallback_reason="insufficient_history"` [历史不足 fallback 原因]。
      - 执行等权 fallback。
    - `if covariance_matrix is singular` [协方差矩阵奇异]：
      - `optimizer_status="failed_singular_covariance"` [协方差奇异导致优化器失败]。
      - `fallback_reason="singular_covariance"` [协方差奇异 fallback 原因]。
      - 执行等权 fallback。
    - `if optimizer_weights contain non_finite value` [优化器输出含非有限权重]：
      - `optimizer_status="failed_non_finite_weights"` [非有限权重导致优化器失败]。
      - `fallback_reason="non_finite_optimizer_weights"` [非有限权重 fallback 原因]。
      - 执行等权 fallback。
    - `if constrained_target_weights invalid` [平台约束后目标权重无效]：
      - `optimizer_status="failed_constraint_projection"` [约束投影失败]。
      - `fallback_reason="constraint_projection_failed"` [约束投影失败 fallback 原因]。
      - 若 fallback 后仍无有效权重，返回 `failed_no_valid_optimizer_result` [无有效优化器结果失败]。
    - `if observation or log_return_window contains execution-only field` [决策输入或优化器窗口含执行期字段]：
      - 训练或 trial 失败，禁止进入 validation、checkpoint selection 或 final test。
- 【关键变量说明】：
  - `asset_action_q_values` [每资产三动作 Q 值]：
    - 类型/范围：`np.ndarray[float]`，shape 为 `[asset_count, 3]`。
    - 取值说明：每个资产对应 exclude、neutral、include 三个动作的 Q 值。
    - 备注：不可用资产强制 exclude，不由 argmax 自由选择。
  - `asset_action` [资产级三分类动作]：
    - 类型/范围：`int`，取值 `0`、`1`、`2`。
    - 取值说明：
      - `0`：`exclude` [不进入候选池]
      - `1`：`neutral` [不直接进入候选池，只可在 include 不足时作为补足候选]
      - `2`：`include` [直接进入候选池]
    - 备注：补足时允许所有非 include 且可用资产参与排序，不限 neutral。
  - `min_selected_assets` [优化器候选池最小资产数量]：
    - 类型/范围：`int`，固定值 `2`。
    - 取值说明：正式口径要求至少两个候选资产进入优化器。
    - 备注：可用资产只有一个时例外，允许 100% 单资产 fallback。
  - `selected_asset_count` [DQN 原始 include 数量]：
    - 类型/范围：`int`，范围 `0` 到 `asset_count`。
    - 取值说明：补足前 DQN 直接选择 include 的资产数量。
    - 备注：不得写成补足后的数量。
  - `optimizer_asset_count` [进入优化器的最终资产数量]：
    - 类型/范围：`int`，范围 `0` 到 `asset_count`。
    - 取值说明：原始 include 加补足后的候选池资产数量。
    - 备注：正式可用资产数不少于 2 时，该值最低为 2。
  - `fill_score` [候选池补足排序分数]：
    - 类型/范围：`float`。
    - 取值说明：`q_include - max(q_neutral, q_exclude)`。
    - 备注：降序选择 top-k。
  - `optimizer_name` [组合优化器名称]：
    - 类型/范围：`str`。
    - 取值说明：`equal_weight`、`markowitz_mean_variance`、`minimum_variance`、`sharpe_maximization`、`risk_parity`。
    - 备注：由 child model id 唯一确定。
  - `DecisionMarketState.log_return_window` [决策时可见收益历史窗口]：
    - 类型/范围：二维收益矩阵，行是 decision date 之前或截至 decision date 可见的历史日期，列是资产。
    - 取值说明：传统优化器的收益、协方差、风险贡献唯一输入窗口。
    - 备注：不得包含 execution date 后数据、next valuation 后收益或 test leakage 字段。
  - `include_count` [include 数量]：
    - 类型/范围：`int`，范围 `0` 到 `asset_count`。
    - 取值说明：原始 DQN 动作等于 include 的资产数量。
    - 备注：正式字段名固定为 `include_count`，不使用 `include_asset_count`。
  - `exclude_count` [exclude 数量]：
    - 类型/范围：`int`，范围 `0` 到 `asset_count`。
    - 取值说明：原始 DQN 动作等于 exclude 的资产数量，包含被不可用 mask 强制 exclude 的资产。
    - 备注：正式字段名固定为 `exclude_count`，不使用 `exclude_asset_count`。
  - `neutral_count` [neutral 数量]：
    - 类型/范围：`int`，范围 `0` 到 `asset_count`。
    - 取值说明：原始 DQN 动作等于 neutral 的资产数量。
    - 备注：正式字段名固定为 `neutral_count`，不使用 `neutral_asset_count`。

      - 测试顶层 `status=completed`，不得为 `partial_diagnostic`，并写 `diagnostic_status=partial_diagnostic` 与 `rankable_in_unified_table=false`。
- 【关键变量说明】：
  - `non_neutral_fill_case` [非 neutral 补足测试场景]：
    - 类型/范围：fixture。
    - 取值说明：include 数小于 2、无 neutral 或 neutral 不足、存在可用 exclude 资产。
    - 备注：断言补足来源允许 exclude。
  - `metric_alias_case` [metric alias 测试场景]：
    - 类型/范围：fixture。
    - 取值说明：同一 result 分别用 alias 和 canonical key 读取，返回相同值。

## 3. 数据契约与接口定义 (Data Contract)

### 3.1 输入参数定义

- `baselines.native_rl.enabled_models` [平台原生模型启用列表]：
  - 类型/必填性：`list[str]` / 按实验配置必填。
  - 校验规则：
    - 元素必须是已注册 model id 或 `hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族 alias]。
    - alias 展开后必须只产生六个 trainable entries 中的 child id。
    - 同一 model id 去重后只运行一次。

- `hpo.trainable_models` [HPO 显式可训练模型列表]：
  - 类型/必填性：`list[str]` / 可选。
  - 校验规则：
    - 若出现 `hybrid_dqn_optimizer_reimplementation` [Hybrid DQN Optimizer 算法族 alias]，必须展开为五个 child。
    - `hpo.native_only=true` [只保留平台原生可训练模型] 时，过滤后的对象必须是六个 trainable entries。
    - alias 不得作为 HPO budget unit。

- `hpo.metric` [HPO 目标指标 key]：
  - 类型/必填性：`str` / HPO 必填。
  - 校验规则：
    - 正式配置使用 `validation_sharpe_minus_drawdown_turnover_penalty` [平台 canonical validation 指标 key]。
    - `validation_penalized_sharpe` [论文需求指标别名] 可被解析，但必须映射为 canonical key。
    - 未知 metric key 按现有错误路径失败。

- `optimizer_name` [Hybrid DQN child 组合优化器名称]：
  - 类型/必填性：`str` / child strategy 必填。
  - 校验规则：
    - 只允许 `equal_weight`、`markowitz_mean_variance`、`minimum_variance`、`sharpe_maximization`、`risk_parity`。
    - 由 child model id 唯一确定，不允许运行时改写为其他值。

- `available_mask_at_decision` [决策时可用资产掩码]：
  - 类型/必填性：`np.ndarray[bool]` / 必填。
  - 校验规则：
    - 长度必须等于资产数。
    - PPO-DQN 和 Hybrid DQN 均必须先应用该 mask 再生成可执行权重。

  - `candidate_weights` [PPO 候选组合权重]：
    - 类型/必填性：`np.ndarray[float]` / PPO-DQN 必填。
    - 校验规则：
      - 长度必须等于资产数。
      - mask 后非负。
      - 可用资产权重和为 1。
      - 若不可归一化、含负值或含非 finite 值，则 `candidate_weights_valid=false` [PPO 候选权重有效标记为 false]。
      - 当 `candidate_weights_valid=false` 且 `hierarchy_action != 4` [高层动作不是可用资产等权 fallback] 时，必须返回 `failed_no_valid_action` [无有效动作失败]，不得静默 fallback。
      - 当 `candidate_weights_valid=false` 且 `hierarchy_action == 4` [高层动作是可用资产等权 fallback] 时，允许可用资产等权执行，但必须写入 `fallback_reason="invalid_candidate_weights_equal_weight"` [候选权重无效后等权 fallback 原因]、`ppo_actor_update_mask=0` [PPO actor 不更新]、`ppo_attribution_weight=0.0` [PPO 收益归因权重为 0]。

- `current_weights` [当前组合权重]：
  - 类型/必填性：`np.ndarray[float]` / PPO-DQN 必填。
  - 校验规则：
    - 长度必须等于资产数。
    - mask 后非负。
    - 可用资产权重和为 1；若不可归一化，则使用可用资产等权作为 current proxy。

- `asset_action_q_values` [每资产三动作 Q 值]：
  - 类型/必填性：`np.ndarray[float]` / Hybrid DQN 必填。
  - 校验规则：
    - shape 必须为 `[asset_count, 3]`。
    - 非 finite Q 值对应资产不得进入 include。
    - 不可用资产强制 exclude。

- `DecisionMarketState.log_return_window` [决策时可见收益历史窗口]：
  - 类型/必填性：二维收益矩阵 / Hybrid DQN 传统优化器 child 必填。
  - 校验规则：
    - 只能包含 decision date 可见历史收益。
    - 不得包含 execution date 后数据、next valuation 后收益、realized cost、realized turnover 或任何 execution-only 字段。
    - `lookback_window=252` [收益/协方差回看窗口] 和 `min_observations=60` [最小观测数] 均从该窗口计算。

- `observation` [PPO-DQN 与 Hybrid DQN 决策输入]：
  - 类型/必填性：环境 observation mapping / 必填。
  - 校验规则：
    - 只允许 decision time 可见字段。
    - 禁止 execution-only 字段进入 PPO decision input、DQN signal input、optimizer decision window。
    - 违反时训练或 trial 失败，不进入 final test。

### 3.2 输出/返回定义

- `PortfolioAction` [组合动作对象]：
  - 正常返回：
    - `target_weights` [目标组合权重]：`np.ndarray[float]`，全资产维度，非负，平台约束后可执行。
    - `rebalance_action` [调仓动作类型]：沿用现有平台语义。
    - `rebalance_intensity` [调仓强度]：沿用现有平台语义。
    - `action_info` [动作附加诊断字段]：包含本次新增 diagnostics 字段。
  - 异常返回：
    - `failed_no_valid_action` [无有效动作失败]：可用资产数为 0。
    - `failed_no_finite_validation_metric` [无有限 validation 指标失败]：validation metric 非 finite。
    - `failed_missing_best_checkpoint` [缺失最佳 checkpoint 失败]：训练后未能产生 best checkpoint。

- `training_result` [训练结果 payload]：
  - 正常返回：
    - `status` [训练状态]：`completed`。
    - `rl_training` [是否真实 RL 训练]：`true`。
    - `platform_native_rl_training` [是否平台原生 RL 训练]：`true`。
    - `proxy_training` [是否代理训练]：`false`。
    - `external_original_implementation` [是否外部原始实现]：`false`。
    - `rankable_in_unified_table` [是否允许进入统一主表排名]：`true`。
    - `is_fitted` [策略是否已完成训练]：`true`。
    - `model_name` [平台运行时模型名]。
    - `paper_model_id` [论文可排名模型标识]：PPO-DQN 固定 `ppo_dqn_hierarchical_reimplementation`；Hybrid DQN 固定为当前 child id，且只允许 `hybrid_dqn_optimizer_equal_weight`、`hybrid_dqn_optimizer_markowitz_mean_variance`、`hybrid_dqn_optimizer_minimum_variance`、`hybrid_dqn_optimizer_sharpe_maximization`、`hybrid_dqn_optimizer_risk_parity`。
    - `child_model_name` [child 模型名]：六个 trainable entries 均固定等于 `paper_model_id`。
    - `baseline_family` [baseline 算法族]：固定 `native_rl_reimplementation`。
    - `training_algorithm` [训练算法标识]：PPO-DQN 固定 `ppo_dqn_hierarchical_reimplementation`；Hybrid DQN 固定 `factorized_dqn_signal_plus_portfolio_optimizer`。
    - `clean_room_reimplementation` [clean-room 复现标记]：`true`。
    - `algorithm_fidelity` [算法忠实度]：PPO-DQN 固定 `platform_adapted`；Hybrid DQN 固定 `platform_adapted`。
    - `dqn_role` [DQN 角色]：PPO-DQN 固定 `high_level_action_selector`；Hybrid DQN 不定义该字段为必填输出契约。
    - `factorized_q` [因子化 Q 标记]：Hybrid DQN 固定 `true`；PPO-DQN 不定义该字段为必填输出契约。
    - `portfolio_level_reward_shared` [组合级 reward 共享标记]：Hybrid DQN 固定 `true`；PPO-DQN 不定义该字段为必填输出契约。
    - `counterfactual_asset_reward` [资产级反事实 reward 标记]：Hybrid DQN 固定 `false`；PPO-DQN 不定义该字段为必填输出契约。

    - `checkpoint_last_path` [最后 checkpoint 路径]。
    - `evaluated_checkpoint_path` [实际评估 checkpoint 路径]：final test 成功时必须等于本 run 加载的 best checkpoint 路径。
    - `best_validation_metric` [最佳验证指标]：由 validation split 的 canonical metric 产生。
  - 异常返回：
    - `failed_missing_train_data` [缺失训练数据失败]。
    - `failed_no_finite_validation_metric` [无有限 validation 指标失败]。
    - `failed_missing_best_checkpoint` [缺失最佳 checkpoint 失败]。
    - `failed_checkpoint_load` [checkpoint 加载失败]。

  - 异常返回：
    - 不参与主排名、不参与正式 seed summary、不参与正式 paired statistics。

- `closest_hybrid_figure_source.csv` [closest-hybrid 图表来源 CSV]：
  - 正常返回：
    - 包含六个 trainable entries 的 paper-level 指标、元数据与 figure 所需字段。
    - 来源只允许平台输出 CSV，不允许混排原论文报告指标。
  - 异常返回：
    - 来源行缺少 `algorithm_fidelity` [算法忠实度]、`baseline_family` [baseline 算法族]、`training_algorithm` [训练算法标识] 或 `rankable_in_unified_table` [是否允许进入统一主表排名] 时，不生成 figure source row。

### 3.3 状态变更

- `strategy.is_fitted` [策略是否已完成训练]：
  - 旧状态：`false` [未训练]。
  - 新状态：`true` [已训练]。
  - 触发条件：
    - `fit()` [训练入口] 完成训练、validation selection 与 best checkpoint 保存。
    - `training_result["status"]` [训练状态] 已按下方规则写为 `completed` [完成]。
    - best checkpoint 存在且 validation metric finite。
    - `strategy.is_fitted` [策略是否已完成训练] 在 `training_result["status"]="completed"` [训练完成] 写入之后置为 `true`，避免与 `training_result.status` [训练结果状态] 互为前置条件。

- `training_result.status` [训练结果状态]：
  - 旧状态：未设置。
  - 新状态：
    - `completed` [训练完成]
    - `failed_missing_train_data` [缺失训练数据失败]
    - `failed_no_valid_action` [无有效动作失败]
    - `failed_no_finite_validation_metric` [无有限 validation 指标失败]
    - `failed_missing_best_checkpoint` [缺失最佳 checkpoint 失败]
    - `failed_checkpoint_load` [checkpoint 加载失败]
    - `failed_training_exception` [训练异常失败]
    - `failed_no_valid_optimizer_result` [无有效优化器结果失败]
    - `deferred_variant` [延期变体]
    - `needs_paper_confirmation` [需要论文细节确认]
  - 触发条件：
    - `completed` [训练完成] 仅在 `fit()` [训练入口] 完成真实 RL 训练、validation best checkpoint 选择、best checkpoint 保存、finite validation metric 校验，且 `rl_training=true` [真实 RL 训练]、`platform_native_rl_training=true` [平台原生 RL 训练]、`proxy_training=false` [非代理训练]、`external_original_implementation=false` [非外部原始实现]、`rankable_in_unified_table=true` [允许进入统一主表排名] 时写入。
    - 任一前置条件失败时，按具体失败原因写入对应 failed status。
    - PPO-DQN 非首版 `dqn_role` [DQN 角色] 变体写 `deferred_variant` [延期变体] 或 `needs_paper_confirmation` [需要论文细节确认]，并写 `rankable_in_unified_table=false` [禁止进入统一主表排名]。

- `rankable_in_unified_table` [是否允许进入统一主表排名]：
  - 旧状态：未设置或沿用现有 metadata。
  - 新状态：
    - `true` [允许排名]：六个 trainable entries 正式成功 run。
    - `false` [禁止排名]：alias、proxy、partial diagnostic、失败 run。
  - 触发条件：baseline metadata 和 run status 合并时写入。

- `diagnostic_status` [诊断运行状态]：
  - 旧状态：未设置。
  - 新状态：
    - `partial_diagnostic` [部分诊断完成]
    - `diagnostic_shared_dqn` [共享 DQN 诊断方案]
  - 触发条件：
    - `partial_diagnostic` [部分诊断完成]：显式 Hybrid DQN 算法族 smoke/diagnostic run 中存在必需 child failed/skipped；算法族顶层 `status` 可为 `completed`，但不得为 `partial_diagnostic`，且必须写 `rankable_in_unified_table=false` [禁止进入统一主表排名]。
    - `diagnostic_shared_dqn` [共享 DQN 诊断方案]：共享一个 DQN 后对五个 optimizer 做 final test。

- `failed_child_model_id` [失败 child 模型标识]：
  - 旧状态：未设置。
  - 新状态：失败或 skipped 的 child model id。
  - 触发条件：Hybrid DQN 算法族 smoke/diagnostic 中任一 optimizer child failed 或 skipped。

- `reason` [失败或诊断原因]：
  - 旧状态：未设置。
  - 新状态：失败状态、partial diagnostic 或 diagnostic shared-DQN 的原因文本。
  - 触发条件：任一 child failed/skipped、checkpoint 加载失败、optimizer 无有效结果、非完整 seed grid、不同预算或 shared-DQN 诊断协议。

- 数据库字段：
  - 无数据库字段变更。
  - 本需求只写本地 CSV、checkpoint 和 in-memory/result payload 字段。

## 4. 异常处理与边界场景 (Exception Handling)

- 场景零 [训练数据缺失]：
  - 代码对应的捕获与处理方案：
    - `fit(train_data, validation_data)` [训练入口] 在构造环境前检查 train split 非空。
    - train split 缺失或为空时返回 `failed_missing_train_data` [缺失训练数据失败]。
    - 不创建 checkpoint，不进入 validation，不进入 `BacktestEngine` [统一回测引擎] final test。
  - 技术结果：
    - `fit_required=true` [策略必须先训练标记] 阻止未成功训练的策略进入回测。

- 场景一 [PPO-DQN 无可用资产]：
  - 代码对应的捕获与处理方案：
    - `_resolve_hierarchy_action()` [层级动作解析函数] 检测 `available_asset_count == 0`。
    - 返回 `failed_no_valid_action` [无有效动作失败]。
    - 不生成非 finite `target_weights` [目标组合权重]。
  - 技术结果：
    - 正式/HPO/rankable run 顶层失败。
    - Hybrid DQN 算法族 smoke/diagnostic run 任一必需 child 失败时，顶层 `status` 可写 `completed` [完成]，但必须写 `diagnostic_status=partial_diagnostic` [部分诊断]、`rankable_in_unified_table=false` [禁止进入统一主表排名]，并禁止进入主表。

- 场景二 [PPO-DQN fallback_equal_weight 执行]：
  - 代码对应的捕获与处理方案：
    - `hierarchy_action == 4` [可用资产等权 fallback] 时生成可用资产等权。
    - `ppo_actor_update_mask=0` [PPO actor 不更新]。
    - `ppo_attribution_weight=0.0` [PPO 收益归因权重为 0]。
    - PPO actor loss 跳过或乘以 0。
    - PPO critic 仍用真实 transition 训练。
    - DQN 仍用真实 transition 更新。
  - 技术结果：
    - 收益归因清晰，避免 fallback 收益错误归入 PPO actor。

- 场景三 [PPO-DQN blend surrogate 归因]：
  - 代码对应的捕获与处理方案：
    - blend 动作使用 `candidate_log_prob * rho` [PPO candidate 对数概率乘归因权重] 作为 actor surrogate。
    - 不尝试计算 blended final weights 的精确 log_prob。
    - 写入 `platform_adapted_surrogate=true` [平台适配 surrogate 标记]。
  - 技术结果：
    - daily diagnostics、training summary、baseline comparison 可追溯 surrogate 口径。

- 场景四 [Hybrid DQN include 不足且无 neutral]：
  - 代码对应的捕获与处理方案：
    - 当 `selected_asset_count < 2` [DQN 原始 include 数量不足] 且 `available_asset_count >= 2` [可用资产不少于 2] 时，从所有非 include 且可用资产补足。
    - 补足排序分数固定为 `q_include - max(q_neutral, q_exclude)` [include 相对非 include 优势分数]。
    - 原始 exclude 资产允许被补足。
  - 技术结果：
    - 消除“只能从 neutral 补足”导致无候选的逻辑断层。

- 场景五 [Hybrid DQN 可用资产只有一个]：
  - 代码对应的捕获与处理方案：
    - 不调用传统优化器。
    - 对唯一可用资产赋 100% 权重。
    - 写入 `fallback_reason="single_available_asset"` [仅单可用资产 fallback 原因]。
  - 技术结果：
    - run 不因最小候选数为 2 而失败。

- 场景六 [Hybrid DQN 无可用资产]：
  - 代码对应的捕获与处理方案：
    - 返回 `failed_no_valid_action` [无有效动作失败]。
    - 不调用传统优化器。
    - 不生成 rankable row。
  - 技术结果：
    - 正式/HPO/rankable run 失败。

- 场景七 [传统优化器失败]：
  - 代码对应的捕获与处理方案：
    - `_compute_optimizer_weights()` [候选池组合权重计算函数] 检查 `optimizer_result.success` [优化器是否成功]。
    - 历史不足时写 `optimizer_status="failed_insufficient_history"` [历史不足导致优化器失败] 与 `fallback_reason="insufficient_history"` [历史不足 fallback 原因]。
    - 协方差奇异时写 `optimizer_status="failed_singular_covariance"` [协方差奇异导致优化器失败] 与 `fallback_reason="singular_covariance"` [协方差奇异 fallback 原因]。
    - 优化器输出非有限权重时写 `optimizer_status="failed_non_finite_weights"` [非有限权重导致优化器失败] 与 `fallback_reason="non_finite_optimizer_weights"` [非有限权重 fallback 原因]。
    - 约束后权重无效时写 `optimizer_status="failed_constraint_projection"` [约束投影失败] 与 `fallback_reason="constraint_projection_failed"` [约束投影失败 fallback 原因]。
    - 候选池非空时，最终 fallback 为候选池等权。
    - 候选池为空且存在可用资产时，最终 fallback 为全部可用资产等权。
    - fallback 后仍无有效权重时返回 `failed_no_valid_optimizer_result` [无有效优化器结果失败]。
  - 技术结果：
    - 回测仍可执行，诊断表可追溯优化器失败。

- 场景八 [validation metric 非 finite]：
  - 代码对应的捕获与处理方案：
    - `objective_metric()` [HPO 目标指标读取函数] 和 strategy validation loop 均检查 `np.isfinite(metric_value)`。
    - 非 finite 时返回 `failed_no_finite_validation_metric` [无有限 validation 指标失败]。
    - 不保存该 checkpoint 为 best checkpoint。
  - 技术结果：
    - HPO trial 失败，不把无效 trial 当作有效候选。

- 场景八补充 [checkpoint 缺失或加载失败]：
  - 代码对应的捕获与处理方案：
    - final test 前检查 `checkpoint_best_path` [最佳 checkpoint 路径] 存在。
    - 缺失时返回 `failed_missing_best_checkpoint` [缺失最佳 checkpoint 失败]。
    - 加载异常时返回 `failed_checkpoint_load` [checkpoint 加载失败]。
    - 加载成功后写 `evaluated_checkpoint_path` [实际评估 checkpoint 路径]，且该路径等于本 run 的 best checkpoint。
  - 技术结果：
    - checkpoint 缺失或加载失败时不得进入 test backtest。

- 场景九 [Hybrid DQN 算法族 alias 被写入 rankable 主表]：
  - 代码对应的捕获与处理方案：
    - `_paper_main_comparison()` [论文主对比表生成函数] 过滤 `model_name` 或 `paper_model_id` 等于 `hybrid_dqn_optimizer_reimplementation` 的行。
    - metadata 层将 alias 行 `rankable_in_unified_table=false`。
  - 技术结果：
    - `paper_main_comparison.csv` [论文主对比表] 不出现算法族总行。

- 场景十 [sidecar diagnostics 缺少 paper_model_id]：
  - 代码对应的捕获与处理方案：
    - 对六个 trainable entries，sidecar row 生成时校验 `paper_model_id` [论文可排名模型标识] 非空。
    - 缺失时将 row 标记不可排名，正式 run 进入失败路径。
  - 技术结果：
    - Hybrid DQN 五个 child 的诊断行不会因 `model_name` [平台运行时模型名] 相同或保留算法族名而歧义。

- 场景十一 [execution-only 字段进入训练或决策输入]：
  - 代码对应的捕获与处理方案：
    - PPO-DQN 在构造 `observation` [PPO-DQN 决策输入] 后执行 decision-visibility 校验。
    - Hybrid DQN 在构造 DQN signal 输入和 `DecisionMarketState.log_return_window` [决策时可见收益历史窗口] 后执行 decision-visibility 校验。
    - 发现 execution date 后数据、next valuation 后收益、realized cost、realized turnover 或其他 execution-only 字段时，训练或 trial 失败。
  - 技术结果：
    - 禁止 test leakage 或 future information leakage。

- 场景十二 [smoke/diagnostic child 失败或 skipped]：
  - 代码对应的捕获与处理方案：
    - 五个 Hybrid optimizer child 均 `completed` [完成] 时，显式 smoke/diagnostic run 顶层 `status=completed` [完成状态]。
    - 任一 Hybrid optimizer child failed 或 skipped 时，显式 smoke/diagnostic run 顶层 `status=completed` [完成状态]，不得写 `partial_diagnostic` [部分诊断] 作为顶层状态。
    - 同时写 `diagnostic_status=partial_diagnostic` [部分诊断状态]、`rankable_in_unified_table=false` [不可进入统一主表排名]、`failed_child_model_id` [失败 child 模型标识]、`reason` [失败原因]。
  - 技术结果：
    - 诊断 run 不误入主表，失败 child 可追溯。

- 系统稳定性：
  - 超时：
    - 训练循环继续使用现有 `max_train_steps` [最大训练步数]、`max_epochs` [最大训练轮数]、HPO trial budget 控制，不新增无限循环。
  - 网络波动：
    - 本需求不依赖网络服务。
  - 文件写入失败：
    - 沿用现有 logger 文件写入异常路径。
    - `metrics/baseline_daily_diagnostics.csv` [baseline 每日诊断 sidecar 文件] 写入失败时，该 run 视为 artifact 不完整，不进入正式 rankable 输出。
  - checkpoint 缺失：
    - strategy final test 前检查 `evaluated_checkpoint_path` [最终评估 checkpoint 路径]。
    - 缺失时返回 `failed_missing_best_checkpoint` [缺失最佳 checkpoint 失败]。

## 6. 非功能性约束 (Constraints)

- 安全与路径约束：
  - `permission_check` [权限校验]：
    - 本需求不新增用户权限模型。
    - 所有输出文件仍写入当前 experiment output root 下的 `metrics/` [指标输出目录]、`logs/` [日志输出目录]、checkpoint 目录。
    - 不允许写出项目运行目录之外的 artifact。
  - 文件路径：
    - `metrics/baseline_daily_diagnostics.csv` [baseline 每日诊断 sidecar 文件] 为正式 sidecar 路径。
    - `metrics/hpo_model_final_daily_diagnostics.csv` [HPO per-model final 诊断 sidecar 文件] 为 HPO 聚合路径。

- 性能约束：
  - PPO-DQN：
    - 新增 DQN hierarchy 网络只输出五个动作，不扩展连续动作空间。
    - PPO candidate 权重生成后仅进行 mask、normalize、blend，不增加复杂二次优化。
  - Hybrid DQN：
    - 每资产 DQN 输出三动作 Q 值，候选池补足排序复杂度为 `O(asset_count log asset_count)`。
    - 传统优化器只在候选池上运行，不在全资产上强制运行。
  - HPO：
    - 算法族 alias 展开为五个 child budget unit。
    - alias 自身不消耗 HPO budget。

- 缓存策略：
  - 不新增全局缓存。
  - 传统优化器可复用现有历史收益窗口计算方式。
  - checkpoint 保存策略沿用现有 native baseline 训练模板。

- 可观测性约束：
  - PPO-DQN 必须记录：
    - `hierarchy_action` [DQN 层级执行动作编号]
    - `hierarchy_action_name` [DQN 层级执行动作名称]
    - `ppo_actor_update_mask` [PPO actor 更新掩码]
    - `ppo_attribution_weight` [PPO 收益归因权重]
    - `platform_adapted_surrogate` [平台适配 surrogate 标记]
  - Hybrid DQN 必须记录：
    - `include_count` [include 数量]
    - `exclude_count` [exclude 数量]
    - `neutral_count` [neutral 数量]
    - `selected_asset_count` [DQN 原始 include 数量]
    - `optimizer_asset_count` [进入优化器的最终资产数量]
    - `optimizer_name` [组合优化器名称]
    - `optimizer_status` [优化器状态]
    - `fallback_reason` [fallback 原因]
  - 诊断字段必须进入 sidecar，不强制进入 `daily_rebalance.csv` [每日调仓明细表]。
