# 强化学习资产配置_实验设计

生成日期：2026-05-15

## 1. 实验目标

目标不是只跑通单个模型，而是验证平台中的深度强化学习资产配置方法在统一数据、统一交易执行、统一成本、统一约束和统一统计检验下是否稳定优于传统资产配置方法与原生深度 RL 基准。

核心问题：

1. `full_dqn_gated_multitask_cnn_ppo` 是否在扣除交易成本后取得更好的风险调整收益。
2. DQN 调仓门控是否能降低换手和成本，并改善净收益指标。
3. CNN 表征、多任务辅助目标、MetricsFactory 特征、PCA、奖励函数设计是否真正贡献了样本外效果。
4. 在不同成本、调仓频率、市场状态、资产池和随机种子下，结果是否稳健。
5. 平台内 native RL baselines 与传统基准相比是否提供有效增量。

成交与收益双数据源规则：

- TuShare `fund_nav.adj_nav` 是训练收益、奖励、风险指标、MetricsFactory 指标和报告收益的主口径。
- 交易所 OHLCV / amount 是 `next_open` 成交、流动性、换手、市场冲击成本和执行约束的主口径。
- 正式 `next_open` 实验必须同时具备 Core-13 的 `adj_nav` 和 OHLCV / amount / vol 字段，并按 `ts_code + date` 内连接对齐。
- 若某产品或某日期缺少任一正式必需字段，应在 strict common history 下剔除该日期；不能用单资产可用区间补齐。
- 若暂时只有 NAV 数据而没有 OHLCV，必须另命名为 `nav_only_execution_proxy` 或等价诊断实验，且不得进入正式主表、成本敏感性或 `next_open` 结论。
- `wide_pct_chg` 和 `wide_log_return` 在 Core-13 正式数据中必须来自 `adj_nav`；`wide_open/high/low/close/pre_close` 保留交易执行价格口径，并在 manifest 中记录口径差异。
- 当前执行核心仍把 `wide_close` 同时用于决策估值、执行前收益、持有收益和 NAV 更新；因此 Core-13 正式 `next_open + adj_nav` 实验还需要先完成估值/收益字段与执行价格字段分离的代码支持。
- 代码支持落地前，只能选择二者之一：保持 `next_open` 但收益为交易价格口径，或降级为 `nav_only_execution_proxy`；不能声称正式结果是 `adj_nav` 收益口径且 `next_open` 执行口径。
- 正式改造目标：环境/回测应从 `adj_nav` 或等价 valuation table 计算组合收益和报告 NAV，从 OHLCV table 读取成交价、流动性和成本观测字段，并在测试中覆盖两套字段不相等的情况。

特征量纲与降维协议：

- 训练收益、奖励、风险指标、MetricsFactory 指标必须基于 TuShare `fund_nav.adj_nav`。
- 原始 `unit_nav` 只能用于数据审计和缺失诊断，不得进入正式训练收益、奖励或指标工厂口径。
- OHLCV 可以作为辅助特征，但必须保留“二级市场交易价格口径”标记，不得替代 `adj_nav` 收益口径。
- OHLC 价格类字段不得以绝对价格直接进入模型；必须先转换为相对量，例如 `open / prev_close - 1`、`close / open - 1`、`high / low - 1`、`close / adj_nav - 1`、折溢价率或滚动收益。
- 成交量、成交额、基金规模、资产净值、份额等重尾规模字段不得以原始数量级直接进入 PCA 或模型；必须先做 `log1p` 或相对化处理，再进入 winsorize / 标准化。
- 费率、换手率、折溢价率、仓位比例等比例字段必须统一为小数口径，例如 `3%` 记为 `0.03`。
- 年化收益、年化波动、Sharpe、Sortino、Calmar、CVaR 等指标必须固定窗口和年化因子；同一指标不得混用自然日和交易日口径。
- PCA / feature selection 前必须完成缺失处理、异常值 winsorize 和标准化；imputer、winsorize、scaler、feature selection、PCA 只能在 train split 上 fit，再应用到 validation / test。
- 不允许把数千维原始指标矩阵直接喂给模型；高维 MetricsFactory + OHLCV 输入必须经过 feature selection 和/或 PCA 后形成低维输入矩阵。默认 `M6` 使用完整特征 + PCA；若原始维度达到千维级，优先评估 `M7` 完整特征 + feature selection + PCA。

选择规则：

- 所有模型选择、HPO、早停、best checkpoint 只能使用 validation split。
- test split 只能做最终报告，不参与任何参数选择。
- 统计检验基于 paired out-of-sample daily net returns。
- `rankable_in_unified_table = false` 的 proxy / external original 结果只能进入诊断表，不进入主排名和主统计检验。

## 4. 实验假设

H1：主模型在 `report_sharpe`、`calmar`、`max_drawdown`、`CVaR` 上优于传统基准与 native RL 基准。

H2：DQN gate 相比无 gate 的 PPO / CNN-PPO 能降低 `turnover`、`total_transaction_cost`，并提升扣费后指标。

H3：多任务辅助目标能改善样本外稳定性，表现为更高的多 seed 均值、更低的 seed 间方差。

H4：`M6 = full features + PCA` 比低维输入和全量未降维输入更稳健。

H5：在高交易成本、高波动、熊市或震荡市中，partial rebalance、uncertainty-aware 和 distributional-CVaR 版本更有优势。

H6：结果在 walk-forward folds 中保持方向一致，而不是由单一固定 split 驱动。

H7：在统一 Core-13 ETF/LOF、`adj_nav` 指标、next-open 执行、成本和 HPO 协议下，主模型的 binary rebalance gate 与 closest-hybrid PPO-DQN / DQN-optimizer 路线相比至少保持 competitive；若 P9 baseline 超过主模型，论文结论必须收窄为“统一协议下的结构比较”，不能写成主模型绝对优势。

## 7. 主指标与统计检验

主指标：

- `report_sharpe = metrics.sharpe`，由 `daily_returns.net_return` 序列计算，代表扣费后收益序列的 Sharpe。
- 若 valuation / return table 尚未与 execution price table 分离，则 `daily_returns.net_return` 只能解释为交易价格收益诊断口径，不能作为正式 `adj_nav` 报告收益。
- 正式 `adj_nav` 口径下，`report_sharpe` 必须基于由 `adj_nav` valuation table 计算出的 `daily_returns.net_return`。

命名边界：

- 报告表、图和主排序使用 `report_sharpe` / `metrics.sharpe`。
- HPO 使用 `validation_penalized_sharpe = validation_sharpe_minus_drawdown_turnover_penalty`。
- 代码中 `objective_metric(..., "net_sharpe")` 是兼容别名，实际计算为 Sharpe - max_drawdown - average_turnover，不得在结果报告里写成普通 Sharpe。

次指标：

- cumulative return
- annualized return
- annualized volatility
- sortino
- calmar
- max_drawdown
- VaR
- CVaR
- turnover
- total_transaction_cost
- violation rate

`violation rate` 必须在正式配置中明确来源。推荐定义为约束管理器或回测日度输出中的
constraint violation 次数 / 可交易决策日数；如果当前 run 没有对应字段，则该指标标记为
`not_available`，不得手工补零。

统计检验：

- paired bootstrap confidence interval
- HAC / Newey-West
- Probabilistic Sharpe Ratio
- Deflated Sharpe Ratio
- White Reality Check
- Hansen SPA
- Diebold-Mariano
- Holm-Bonferroni 多重检验校正

统计边界：

- 主统计检验只使用 paired out-of-sample daily net returns。
- `rankable_in_unified_table = false` 的 proxy / external original 结果必须排除。
- 若 `run_manifest.data_governance.return_source != "adj_nav"`，或 `run_manifest.execution_model.valuation_execution_split != true`，该 run 不得进入 Table A / Table G / Table H 的正式排名和正式 paired tests。
- `paper_aggregate` 必须二次过滤诊断 run；不得只依赖上游实验命名。
- deterministic baselines 不得通过复制 seed 伪造独立样本；seed summary 必须区分 `n_seeds` 与 `n_unique_return_series`。
- Diebold-Mariano 仅在存在可比 forecast error 序列时进入主统计表；否则允许记录为 skipped，不视为实验失败。
- White Reality Check / Hansen SPA 用于多策略数据挖掘校正，主文只报告与核心结论相关的比较，完整结果放附录。
- 如果 `statistics_summary.csv` 大面积 `not_applicable`，该 run 不能进入论文主表。

主比较：

- 主模型 vs 最强传统基准。
- 主模型 vs `cnn_ppo_native`。
- 主模型 vs `pgportfolio_eiie_native`。
- 主模型 vs without gate / without auxiliary / no PCA。
- 主模型 vs P9 六个 closest-hybrid clean-room reimplementation。
- `ppo_dqn_hierarchical_reimplementation` vs `ppo_native`。
- `hybrid_dqn_optimizer_sharpe_maximization` vs `dqn_template_native`。

P9 必做 paired tests：

```text
full_dqn_gated_multitask_cnn_ppo vs ppo_dqn_hierarchical_reimplementation
full_dqn_gated_multitask_cnn_ppo vs hybrid_dqn_optimizer_equal_weight
full_dqn_gated_multitask_cnn_ppo vs hybrid_dqn_optimizer_markowitz_mean_variance
full_dqn_gated_multitask_cnn_ppo vs hybrid_dqn_optimizer_minimum_variance
full_dqn_gated_multitask_cnn_ppo vs hybrid_dqn_optimizer_sharpe_maximization
full_dqn_gated_multitask_cnn_ppo vs hybrid_dqn_optimizer_risk_parity
ppo_dqn_hierarchical_reimplementation vs ppo_native
hybrid_dqn_optimizer_sharpe_maximization vs dqn_template_native
```

P9 统计解释：

- 所有统计都用 paired out-of-sample daily net returns。
- deterministic baselines 只生成一条唯一 daily returns 序列；paired tests 可与每个 stochastic seed 分别配对。
- seed summary 中 deterministic baseline 必须报告 `n_unique_return_series = 1`。
- 不得把复制出的 5 条 identical seed 序列当成 5 个独立样本计算 std / standard error。
- Holm correction 后不显著时，论文不能写成显著优于。
- 若 P9 新 baseline 优于主模型但不显著，应写成 competitive / comparable，不能写 dominates。

## 9. 执行顺序建议

5. P7 formal：正式 equal-budget HPO，生成 best config / best checkpoint。
6. P1 final：使用 HPO 后配置生成最终主对比。

14. P8：preference / uncertainty / distributional / partial 模块。

实验监控频率：

- 正式实验可能耗时较久，实验启动后智能体无需时刻盯着终端。
- 超长实验：例如 `FULL_v1`、完整 walk-forward、大规模 HPO，启动后每 10 分钟检查一次进度。
- 长实验：例如 baseline comparison、reward ablation、input matrix ablation、PCA ablation、cost sensitivity，启动后每 5 分钟检查一次进度。
- 短实验：例如 P0 smoke、单 seed main model、小规模配置校验，启动后每 1 分钟检查一次进度。
- 每次检查只需确认进程是否仍在运行、日志是否持续输出、是否出现 failed / traceback / NaN / Inf / out-of-memory。
