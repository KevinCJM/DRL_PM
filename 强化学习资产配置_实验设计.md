# 强化学习资产配置_实验设计

生成日期：2026-05-15

## 1. 实验目标

目标不是只跑通单个模型，而是验证平台中的深度强化学习资产配置方法在统一数据、统一交易执行、统一成本、统一约束和统一统计检验下是否稳定优于传统资产配置方法与原生深度 RL 基准。

核心问题：

1. `full_dqn_gated_multitask_cnn_ppo` 是否在扣除交易成本后取得更好的风险调整收益。
2. DQN 调仓门控是否能降低换手和成本，并改善净收益指标。
3. CNN 表征、多任务辅助目标、MetricsFactory 特征、PCA、奖励函数设计是否真正贡献了样本外效果。
4. 在不同成本、调仓频率、市场状态、资产池和随机种子下，结果是否稳健。
5. 平台内 native RL baselines 与传统基准相比是否提供有效增量。

## 2. 固定实验协议

默认数据：

- 数据目录：`data/processed/`
- 指标目录：`data/metrics_factory/`
- 资产池：`data/processed/asset_universe.csv` 中 `status == "ok"` 的资产。
- 默认特征：`feature_matrix.input_matrix_id = M6`
- 默认 PCA：`feature_reduction.pca.enabled = true`，`explained_variance = 0.95`
- 默认成交：`execution_model.execution_price = next_open`
- 默认无前视：`execution_model.strict_no_lookahead_execution = true`
- 默认组合资金规模：`portfolio.initial_capital_currency = 100000000.0`，`currency = CNY`
- 默认交易成本：`proportional_cost = 0.001`，`slippage = 0.0005`，`market_impact_enabled = true`
- 默认再平衡：`rebalance.mode = monthly`
- 默认主奖励：`reward.mode = A2_net_log_return_after_cost`

选择规则：

- 所有模型选择、HPO、早停、best checkpoint 只能使用 validation split。
- test split 只能做最终报告，不参与任何参数选择。
- 统计检验基于 paired out-of-sample daily net returns。
- `rankable_in_unified_table = false` 的 proxy / external original 结果只能进入诊断表，不进入主排名和主统计检验。

## 4. 实验假设

H1：主模型在 `report_sharpe`、`calmar`、`max_drawdown`、`CVaR` 上优于传统基准与 native RL 基准。

H2：DQN gate 相比无 gate 的 PPO / CNN-PPO 能降低 `turnover`、`total_transaction_cost`，并提升扣费后指标。

H3：多任务辅助目标能改善样本外稳定性，表现为更高的多 seed 均值、更低的 seed 间方差。

H4：`M6 = full features + PCA` 比低维输入和全量未降维输入更稳健。

H5：在高交易成本、高波动、熊市或震荡市中，partial rebalance、uncertainty-aware 和 distributional-CVaR 版本更有优势。

H6：结果在 walk-forward folds 中保持方向一致，而不是由单一固定 split 驱动。

### P1 主模型与基准对比

目的：建立主结论。

实验：

推荐正式预算：

- 单次命令默认只运行一个 seed。正式 5-seed 结果需要分别运行 `--seed 42/123/2024/3407/9999` 后再聚合，或使用专门的 `seed_stability` 实验。

- primary report metric：`report_sharpe = metrics.sharpe`，由 `daily_returns.net_return` 序列计算。
- HPO objective：`validation_penalized_sharpe = validation_sharpe_minus_drawdown_turnover_penalty`。
- secondary metrics：`calmar`、`max_drawdown`、`cvar`、`turnover`、`total_transaction_cost`

### P3 算法组件消融

目的：拆解主模型增量来源。

建议实验组与当前执行状态：

- full model：`configs/experiments/main.yaml`
- without DQN gate：`configs/paper/ablation_without_dqn_gate.yaml`，`dqn.enabled=false` 必须使 rollout / 回测动作始终走 PPO candidate rebalance，不创建 DQNAgent，不用 gate Q 决策调仓。
- without auxiliary heads：`configs/paper/ablation_without_auxiliary.yaml`，`auxiliary.enabled=false` 必须不创建辅助头，HybridAgent 辅助更新返回 `not_configured`。
- without CNN encoder / use MLP encoder：`configs/paper/ablation_mlp_encoder.yaml`，override `model.default_encoder=mlp`、`model.encoder.type=mlp`
- cross-asset attention diagnostic：`configs/paper/ablation_attention_enabled.yaml`，override `model.encoder.cross_asset_attention.enabled=true`
- without PCA：使用 `configs/paper/pca_ablation.yaml` 中的 `no_pca` variant
- DQN gate only / template selector：使用 `dqn_template_native` baseline，不作为同一个 ablation YAML
- PPO continuous allocation only：使用 `ppo_native` 或 `cnn_ppo_native` baseline，不作为同一个 ablation YAML
- without market impact cost in training but real-cost evaluation：当前没有独立 train/eval cost split 配置，不进入正式主消融；如需执行，应先补代码配置合同。

当前可作为冻结配置的入口：

```bash

.venv/bin/python -m src.experiments.run_experiment --config configs/paper/ablation_without_auxiliary.yaml --run-name P3_without_auxiliary
.venv/bin/python -m src.experiments.run_experiment --config configs/paper/ablation_mlp_encoder.yaml --run-name P3_mlp_encoder
```

`configs/experiments/auxiliary_task_sensitivity.yaml` 是矩阵诊断入口；论文正式 P3 辅助头消融优先使用 `configs/paper/ablation_without_auxiliary.yaml`，并与外部 full model run 聚合对比。

正式消融必须满足单开关原则：

- 每个 ablation run 只改变一个组件或一个同族参数。
- 每个 run 必须记录 `ablation_id`、`changed_key_path`、`base_run_id`。
- 不得把 `without DQN gate`、`without auxiliary heads`、`without PCA` 合并到同一个 run。
- `configs/paper/ablation_*.yaml` 当前是“单独 ablated run”配置，不是自动 full-vs-ablation 双臂实验。
- generic `experiment.type=ablation` 的 `base/current` 机制当前会继承同一个 active config；因此这些 run 的对照组必须来自外部已完成的 full model run，例如 `EXP01_main` 或 HPO final full-model run。
- 生成论文表时必须把 full run 与 ablated run 的 `daily_returns.csv` / `metrics.csv` 外部合并比较，不得直接把 generic ablation 的 `base/current` 当作有效消融对比。
- `ablation_results.csv` 不能直接作为论文消融表。论文消融表必须按第 8 节聚合协议，从 full run 与 ablated run 的 daily / metrics / comparison 产物外部合并生成。

如果某个组件当前没有明确配置键，先标记为 `needs_code_confirmation`，不要用临时手改配置替代论文实验。

### P7 HPO 公平实验

目的：避免“主模型调参更多、基准调参更少”的不公平比较。

运行：

```bash
.venv/bin/python -m src.experiments.run_experiment --config configs/paper/hpo_equal_budget_native_pilot.yaml --run-name P7_hpo_pilot
.venv/bin/python -m src.experiments.run_experiment --config configs/paper/hpo_equal_budget_native.yaml --run-name P7_hpo_equal_budget_native
```

正式配置建议：

```yaml
hpo:
  enabled: true
  equal_budget_across_models: true
  native_only: true
  n_trials_per_model: 50
  metric: validation_sharpe_minus_drawdown_turnover_penalty
  selection_split: validation
  final_report_split: test
  trainable_models:
    - full_dqn_gated_multitask_cnn_ppo
    - ppo_native
    - cnn_ppo_native
    - bernoulli_gated_ppo_native
    - dqn_template_native
    - eiie_native
    - pgportfolio_eiie_native
```

报告要求：

- 每个模型相同 `n_trials_per_model`。
- 输出 best / median / worst validation trial 的 test 表现。
- HPO 表必须有 `model_name`、`fold_id`、`trial_number`、`params_json`、`validation_metric`。

注意：`configs/experiments/hyperparameter_sweep.yaml` 只是轻量入口，不是正式公平 HPO 配置。
正式实验使用 `configs/paper/hpo_equal_budget_native.yaml`；开正式大规模 HPO 前，先用
`configs/paper/hpo_equal_budget_native_pilot.yaml` 验证闭环。冻结配置必须显式写入：

- `hpo.equal_budget_across_models = true`
- `hpo.native_only = true`
- `hpo.n_trials_per_model`
- `hpo.trainable_models`
- `training.epochs` 或等价训练步数
- `baselines.native_rl.epochs`
- `baselines.native_rl.max_train_steps`、`max_validation_steps`、`max_gradient_updates_per_epoch`
- `reproducibility.seeds`
- `output.run_name`

公平 HPO 的主表约束：

- 主排名中的所有 trainable models 必须使用完全相同的 HPO trial 数、seed protocol、验证指标和选择规则。
- 如果某个模型使用更大预算，其结果只能进入 robustness / appendix，不能进入主排名。
- walk-forward + HPO 同时启用时，每个 fold 内独立 HPO；不得跨 fold 复用 best trial。
- 单次 HPO 命令只使用一个 sampler seed。正式 5-seed HPO 需要分别用 `--seed` 或独立配置运行 5 次，再聚合 `hpo_trials.csv` 和 final test 结果。
- equal-budget HPO 父结果顶层只代表 best model 的 final payload；其他模型通常只在 `hpo_model_results` / 子 HPO run / `hpo_trials.csv` 中可追溯。Table A 不得直接把父级 `metrics/*` 当成所有模型的最终主表。
- HPO 后生成 Table A 有两种合法方式：
  - 方式 1：读取每个模型子 HPO run 的 best config / best checkpoint，再逐模型重新执行 final backtest，形成统一 `paper_main_comparison.csv`。
  - 方式 2：从每个模型子 HPO run 中提取其 final test daily returns 和 metrics；如果缺少 per-model daily returns，则该模型不能进入 HPO 后主表。

## 7. 主指标与统计检验

主指标：

- `report_sharpe = metrics.sharpe`，由 `daily_returns.net_return` 序列计算，代表扣费后收益序列的 Sharpe。

命名边界：

- 报告表、图和主排序使用 `report_sharpe` / `metrics.sharpe`。
- HPO 使用 `validation_penalized_sharpe = validation_sharpe_minus_drawdown_turnover_penalty`。
- 代码中 `objective_metric(..., "net_sharpe")` 是兼容别名，实际计算为 Sharpe - max_drawdown - average_turnover，不得在结果报告里写成普通 Sharpe。

次指标：

- cumulative return
- annualized return
- annualized volatility
- sortino
- calmar
- max_drawdown
- VaR
- CVaR
- turnover
- total_transaction_cost
- violation rate

`violation rate` 必须在正式配置中明确来源。推荐定义为约束管理器或回测日度输出中的
constraint violation 次数 / 可交易决策日数；如果当前 run 没有对应字段，则该指标标记为
`not_available`，不得手工补零。

统计检验：

- paired bootstrap confidence interval
- HAC / Newey-West
- Probabilistic Sharpe Ratio
- Deflated Sharpe Ratio
- White Reality Check
- Hansen SPA
- Diebold-Mariano
- Holm-Bonferroni 多重检验校正

统计边界：

- 主统计检验只使用 paired out-of-sample daily net returns。
- `rankable_in_unified_table = false` 的 proxy / external original 结果必须排除。
- Diebold-Mariano 仅在存在可比 forecast error 序列时进入主统计表；否则允许记录为 skipped，不视为实验失败。
- White Reality Check / Hansen SPA 用于多策略数据挖掘校正，主文只报告与核心结论相关的比较，完整结果放附录。
- 如果 `statistics_summary.csv` 大面积 `not_applicable`，该 run 不能进入论文主表。

主比较：

- 主模型 vs 最强传统基准。
- 主模型 vs `cnn_ppo_native`。
- 主模型 vs `pgportfolio_eiie_native`。
- 主模型 vs without gate / without auxiliary / no PCA。

论文表格聚合协议：

- Table A 不直接等于某一个 run 的单个 CSV。它需要聚合主模型 run、baseline comparison run、HPO final run 的结果。
- 聚合输入：
  - `results/<main_run>/metrics/daily_returns.csv`
  - `results/<main_run>/metrics/main_comparison.csv`
  - `results/<main_run>/metrics/metrics.csv`
  - `results/<baseline_run>/metrics/baseline_comparison.csv`
  - `results/<baseline_run>/metrics/daily_returns.csv`
  - `results/<hpo_run>/logs/hpo_trials.csv`
  - `results/<hpo_run>/metrics/hpo_model_final_reports.csv`
  - `results/<hpo_run>/metrics/hpo_model_final_comparison.csv`
  - `results/<hpo_run>/metrics/hpo_model_final_daily_returns.csv`
  - `results/<hpo_run>/logs/config_snapshot.yaml`
  - 每个模型子 HPO run 的 best config / best checkpoint / final test daily returns。
  - 注意：`main_comparison.csv` / `baseline_comparison.csv` 是 `model_name`、`rankable_in_unified_table`、baseline family、训练协议等主表元数据的优先来源；`metrics.csv` 只作为补充指标源。
  - 注意：equal-budget HPO 父级 `metrics/*` 只能代表 best model 顶层 payload，不能替代 per-model final daily returns。
- 聚合过滤：
  - 只纳入 `rankable_in_unified_table = true`。
  - 排除 neural proxy 和 `pgportfolio_original_external` 的主排名行。
  - 同一模型多 seed 时先按 seed 计算指标，再报告 mean / std / median / best / worst。
- 输出目标：
  - `paper_main_comparison.csv`
  - `paper_paired_statistics.csv`
  - `paper_seed_summary.csv`
- 成对统计必须基于同一测试日期上的 paired daily net returns；不同 run 的日期不完全一致时，先 inner join 日期。

聚合入口：

```bash
.venv/bin/python -m src.experiments.paper_aggregate \
  --run-dir results/EXP01_main \
  --run-dir results/EXP04_baselines_native \
  --run-dir results/EXP09_hpo_equal_budget_native \
  --benchmark-model best_traditional \
  --benchmark-model cnn_ppo_native \
  --benchmark-model pgportfolio_eiie_native \
  --paper-group-id main_fixed \
  --output-dir results/paper_tables
```

`paper_aggregate` 默认使用论文协议 benchmark 列表；如需限定统计对象，可重复传入
`--benchmark-model`。`paper_seed_summary.csv` 默认只汇总论文指标白名单，避免把
`trial_number`、`best_value` 等运行元数据混入 seed 汇总。聚合器会生成
`paper_model_id`，用于区分 `full_model`、`without_dqn_gate`、`without_auxiliary`、
`no_pca` 等论文口径标签；未设置 `--paper-group-id` 时，统计分组按
`source_run + seed + fold_id` 隔离，避免把不同实验的同名 `model_name` 合并。若需要
跨 run 做同一主表内的成对统计，例如 `main_model + baseline_comparison_native`，
必须显式设置同一个 `--paper-group-id`，此时统计分组按
`paper_group_id + seed + fold_id` 执行。
HPO run 若存在 `hpo_model_final_comparison.csv`，聚合器只读取该 per-model final 表，
避免 best payload 的 `main_comparison.csv` 与 HPO final 表重复进入主表。
equal-budget HPO 还必须落盘 `hpo_model_final_reports.csv`，逐模型记录
best / median / worst validation trial 在 final split 上的测试表现；缺失该表时，
P7 只能视为流程验证，不能作为论文 HPO 审计表来源。

## 9. 执行顺序建议

推荐顺序：

1. P0：pytest + main smoke + baseline smoke。
2. P7 pilot：小规模 equal-budget HPO，确认多模型 HPO 闭环。
3. P1 pilot：主模型 + baseline comparison，确认所有模型可训练和回测。
4. P7 formal：正式 equal-budget HPO，生成 best config / best checkpoint。
5. P1 final：使用 HPO 后配置生成最终主对比。
6. P2：输入矩阵和 PCA。
7. P3：算法组件消融。
8. P4：奖励消融。
9. P5：成本和调仓频率。
10. P6：seed、market regime、asset universe、walk-forward。
11. P8：preference / uncertainty / distributional / partial 模块。
12. FULL：平台完整回归归档。

不建议先跑 FULL，因为任何一个子实验失败都会浪费大量时间。

## 10. 资源预算

CPU 可执行：

- P0 smoke。
- 单 seed、小 epoch 的 P1 / P2 局部实验。

Mac MPS 或单张 GPU 推荐：

- P1 正式 baseline comparison。
- P2 到 P5 消融。
- 5 seed 稳定性。

高配 GPU / 长时间运行推荐：

- HPO 50 trials per model。
- 完整 walk-forward。
- FULL_v1 平台完整回归。

建议磁盘：

- 至少预留 50GB。
- `data/metrics_factory/all_metrics_features.parquet` 约 1GB，实验结果和 checkpoint 会继续增长。

实验监控频率：

- 正式实验可能耗时较久，实验启动后智能体无需时刻盯着终端。
- 超长实验：例如 `FULL_v1`、完整 walk-forward、大规模 HPO，启动后每 10 分钟检查一次进度。
- 长实验：例如 baseline comparison、reward ablation、input matrix ablation、PCA ablation、cost sensitivity，启动后每 5 分钟检查一次进度。
- 短实验：例如 P0 smoke、单 seed main model、小规模配置校验，启动后每 1 分钟检查一次进度。
- 每次检查只需确认进程是否仍在运行、日志是否持续输出、是否出现 failed / traceback / NaN / Inf / out-of-memory。

## 11. 冻结标准

一轮实验可以进入结果分析，需要同时满足：

1. 所有目标 `experiment_result.status == "completed"`，且 `run_manifest.status == "success"`、`run_registry.status == "success"`。
2. `run_manifest.json` 记录 execution model、data governance、portfolio currency 和 capital。
3. daily output 中 `nav`、`net_return`、`weights` 无 NaN / Inf。
4. 权重满足 long-only、simplex、availability mask。
5. `statistics_summary.csv` 不是大面积 `not_applicable`。
6. proxy / external original 没有进入主排名统计。
7. HPO 只用 validation 选择，test 只做 final report。
8. walk-forward 每个 fold 都有独立输出和 fold_id。
9. baseline comparison 主表只使用 `rankable_in_unified_table = true` 的模型。
10. HPO / ablation / sensitivity 的正式配置快照必须归档，不能只依赖命令行临时覆盖。
