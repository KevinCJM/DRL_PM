## 0. 设计结论

P12/P13 不重置 Core-13 v2 的数据、执行、成本、约束和评估协议，只新增可训练模型族，并作为 post-hoc model-extension addendum 单独记录。

- P12：`CAGE-EIIE` 是主线候选。它复用 EIIE 的 allocation alpha，并用 cost-aware multilevel gate 控制调仓强度。
- P13：`GT-RCPO-lite` 是探索候选。它验证 graph/transformer/risk-constrained policy 是否值得进入后续 formal。
- P14：只有通过 validation-only promotion gate 的 P12/P13 变体才能进入 formal combined comparison。

P12/P13 的所有 formal 结果必须满足：

- `protocol_id = core13_v2_full_reset_20260522`
- `model_extension_id = core13_v2_p12_p13_20260524`
- `post_hoc_development_disclosure = true`
- `data_cutoff_date = 2026-05-20`
- `rankable_in_unified_table = true` 仅允许 formal run 写入
- smoke / pilot / scout run 必须写 `rankable_in_unified_table = false`

## 2. 通用执行语义

P12/P13 必须沿用平台执行核心，不自行拼 daily returns。

当前 `BaseStrategy.compute_target_weights()` 没有 `scheduler_allowed` 入参。P12/P13 必须先实现一个向后兼容的 scheduler context 扩展：

- `BacktestEngine` / training env 在 action selection 前计算 `scheduler_allowed_rebalance`。
- 若 strategy 实现 `set_decision_context(...)` 或 `compute_target_weights_with_context(...)`，engine/env 必须把 `scheduler_allowed_rebalance`、`first_trade`、`decision_date` 写入该 context。
- P12/P13 strategy 必须设置 `requires_daily_diagnostics=True`，否则 `BacktestEngine` 会在 scheduler-blocked day 直接短路为 hold，strategy 不会被调用，也无法记录 forced-hold replay / diagnostics。
- 若该 context 扩展未实现，P12/P13 只能 smoke，不得进入 pilot/formal。

获得 scheduler context 后，`compute_target_weights()` 或等价 context-aware 方法返回：

```python
PortfolioAction(
    target_weights=candidate_weights,
    rebalance_action=1 if scheduler_allowed and rho > 0 else 0,
    rebalance_intensity=rho,
    action_info={...},
)
```

关键约束：

- `target_weights` 是候选组合权重，不是 `(1-rho)*current + rho*candidate` 后的混合权重。
- `rebalance_intensity = rho` 只交给 execution core 做一次插值。
- 非 scheduler allowed 调仓日强制 `rho=0`、`rebalance_action=0`。
- observation 使用 `decision_time_current_weights`。
- `pre_execution_drifted_weights` 只允许在 execution core 内部计算，或作为事后日志字段，不得进入模型输入、gate 输入、actor 输入或 HPO selection。

这样与执行核心保持一致：

```text
execution_weights =
  pre_execution_drifted_weights
  + rho * (projected_candidate_weights - pre_execution_drifted_weights)
```

## 3. P12：CAGE-EIIE

### 3.1 模型名称

正式可注册模型：

```text
cage_eiie_frozen_gate
cage_eiie_multilevel_gate
cage_eiie_distributional
cage_eiie_joint_light
```

ablation / diagnostic 可注册模型：

```text
cage_eiie_fixed_rho_25
cage_eiie_fixed_rho_50
cage_eiie_fixed_rho_75
```

fixed-rho 变体默认只进入 P12.3 ablation / appendix diagnostic。若要进入 P14 formal，必须和其他模型一样满足 formal manifest、预算、validation selection 和 rankability 门禁。

首批建议顺序：

1. `cage_eiie_frozen_gate`
2. `cage_eiie_multilevel_gate`
3. `cage_eiie_distributional`
4. `cage_eiie_joint_light`

`joint_light` 只有在前两个版本表现稳定后再进入 pilot。

### 3.2 核心思想

EIIE 负责生成候选权重：

```text
candidate_weights_t = EIIE(s_t, previous_weights_t, availability_mask_t)
```

CAGE gate 负责选择调仓强度：

```text
rho_t ∈ {0.00, 0.25, 0.50, 0.75, 1.00}
```

执行核心负责真正成交权重：

```text
executed_weights_t = execution_core(candidate_weights_t, rho_t)
```

不得在模型侧预先计算 `target = (1-rho)*current + rho*candidate` 后再传 `rho`。

### 3.3 组件设计

#### EIIE Candidate Expert

实现方式：

- 复用 `NativeEIIEStrategy` 的 evaluator / PVM / execution-aligned training sample 逻辑。
- 输出 masked softmax 权重。
- 使用 decision-date 可见 `market_image`、`previous_weights`、`availability_mask`。

训练预算：

- CAGE 是一个完整 trainable model。
- EIIE warm-up、gate training、joint fine-tune 的 env steps / gradient updates 必须计入同一个 per-model budget。
- 不允许把 EIIE expert 作为免费预训练资源进入主表。

#### CAGE Gate Network

输入：

```text
decision_time_current_weights
candidate_weights
candidate_delta_weights
estimated_turnover
estimated_cost
rolling_volatility_features
rolling_correlation_risk_features
liquidity_features
current_drawdown
recent_net_return_distribution_features
availability_mask
scheduler_allowed_rebalance
```

输出：

```text
Q(rho=0.00)
Q(rho=0.25)
Q(rho=0.50)
Q(rho=0.75)
Q(rho=1.00)
```

日志字段：

```text
gate_action_index
rho
rebalance_intensity
rebalance_values
estimated_turnover
estimated_cost
scheduler_allowed_rebalance
forced_hold_reason
```

`gate_action` 可继续保留二值含义：

```text
gate_action = 0 if rho == 0 else 1
```

五档动作索引必须使用 `gate_action_index`。

训练 env 当前只保留有限 action_info 字段。P12/P13 formal 前必须二选一：

- 扩展 training env action_info whitelist，使 `gate_action_index`、`rho`、`rebalance_values`、`forced_hold_reason` 可进入 `info` / replay；
- 或者不依赖 env `info` 保存这些字段，改由 strategy / trainer 同步写入 `baseline_daily_diagnostics.csv` sidecar。

不管采用哪种方式，formal run 的 backtest diagnostics sidecar 必须包含这些字段。

#### Distributional Risk Critic

仅 `cage_eiie_distributional` 启用。

输出：

```text
quantile_values[batch, n_quantiles]
CVaR_loss_5
```

CVaR 统一采用 loss 口径：

```text
CVaR_loss_5 >= 0
penalty_cvar = max(0, CVaR_loss_5 - cvar_loss_budget)
```

禁止使用 `abs(CVaR_5)` 这类符号不清的表达。

### 3.4 Reward / Utility

训练 reward 默认使用平台 after-cost net return，不重复扣交易成本：

```text
base_reward_t = net_log_return_after_cost_t
```

额外项只作为偏好或约束：

```text
reward_t =
    base_reward_t
  - lambda_turnover * realized_turnover_t
  - lambda_cvar * max(0, CVaR_loss_5_t - cvar_loss_budget)
  - lambda_dd * drawdown_increase_t
```

Promotion utility 使用固定配置：

```text
utility =
    cumulative_return
  - alpha_mdd * max_drawdown_loss
  - alpha_cvar * CVaR_loss_5
  - alpha_turnover * average_turnover
  - alpha_cost * total_transaction_cost
```

`max_drawdown_loss` 统一为非负 loss 口径：

```text
max_drawdown_loss = abs(max_drawdown)
```

若底层指标已经输出非负 drawdown loss，仍按同名正数使用；不得把负数 `max_drawdown` 直接代入 utility。

默认：

```text
alpha_mdd = 0.25
alpha_cvar = 0.25
alpha_turnover = 2.0
alpha_cost = 10.0
```

如需变更，必须写入 config snapshot 和 search-space manifest。

### 3.5 训练流程

伪代码：

```text
for seed in seeds:
  build train / validation / test artifacts under Core-13 v2 protocol

  initialize EIIE candidate expert
  initialize CAGE gate
  initialize optional distributional critic

  for trial in HPO trials:
    reset budget counters

    if variant uses EIIE warm-up:
      train EIIE expert within trial budget

    for epoch in training epochs:
      for decision step t:
        read decision-visible state
        compute candidate_weights from EIIE expert
        estimate turnover/cost from decision-visible fields

        if scheduler disallows rebalance:
          rho = 0
          rebalance_action = 0
          forced_hold_reason = scheduler_not_allowed
        else:
          rho = gate.select_action(...)
          rebalance_action = 1 if rho > 0 else 0

        env.step(candidate_weights, rebalance_action, rho)
        store transition
        update gate after warmup
        update optional risk critic

    run validation through BacktestEngine
    compute validation_penalized_sharpe and promotion utility
    save best / last checkpoint

  load best checkpoint before final test
  final test must run through BacktestEngine
```

## 4. P13：GT-RCPO-lite

### 4.1 模型名称

正式注册名：

```text
graph_transformer_risk_constrained_actor_critic_lite
```

简称：

```text
gt_rcpo_lite
```

P13 是 scout，不默认进入 P14 formal。只有通过 promotion gate 后才进入 formal comparison。

### 4.2 架构

P13-lite 只保留必要组件：

```text
market image
  -> temporal encoder
  -> rolling-correlation graph attention
  -> simplex actor candidate weights
  -> option gate rho
  -> execution core
```

组件：

- Temporal encoder：轻量 Transformer / temporal attention。
- Graph encoder：基于 rolling correlation 的 asset graph。
- Actor：输出 long-only simplex candidate weights。
- Option gate：输出 `rho ∈ {0, 0.25, 0.50, 1.00}`。
- Critic：value critic。
- Constraint module：turnover / cost Lagrangian。

不在 lite 版本中启用：

- 多层重型 Graph Transformer。
- 多个动态边类型联合训练。
- 复杂 cash / risk-off 子策略。
- 多目标重型 distributional actor-critic。

### 4.3 Graph / Correlation Leakage Guard

rolling correlation 只能使用 decision-date 前可见窗口：

```text
correlation_window = [t-lookback+1, t]
```

禁止：

```text
full_sample_correlation
test_window_correlation
post_hoc_regime_label
future_returns
execution-date unavailable fields
```

graph edge、attention bias、risk features 都必须可由 decision-date 数据重建。

### 4.4 Actor / Gate 执行语义

Actor 输出：

```text
candidate_weights
```

Gate 输出：

```text
rho ∈ {0.00, 0.25, 0.50, 1.00}
```

执行：

```text
PortfolioAction(
  target_weights=candidate_weights,
  rebalance_action=1 if scheduler_allowed and rho > 0 else 0,
  rebalance_intensity=rho,
)
```

与 P12 一样，禁止模型侧预混合权重。

### 4.5 Constraint / Lagrangian

目标约束：

```text
average_turnover <= turnover_budget
total_transaction_cost <= cost_budget
CVaR_loss_5 <= cvar_loss_budget
```

Lagrangian 更新：

```text
lambda_i = max(0, lambda_i + eta_lambda * (constraint_value_i - budget_i))
```

约束指标必须来自 validation split 的 BacktestEngine outputs 或训练 env 的同口径 realized fields，不使用训练 reward 伪指标。

### 4.6 P13 Promotion Gate

P13 必须满足所有条件才可进入 formal：

```text
failed_trial_rate <= 0.20
finite_artifact_rate == 1.0
validation_cumulative_return >= max(reference_main, reference_eiie) - 0.01
validation_total_transaction_cost <= 1.20 * min(reference_main_cost, reference_eiie_cost)
validation_max_drawdown_loss <= 1.10 * min(reference_main_max_drawdown_loss, reference_eiie_max_drawdown_loss)
validation_metric_finite_rate == 1.0
```

不得使用“训练过程无明显崩溃”这类主观条件作为 promotion gate。

机器口径：

```text
failed_trial_rate = failed_trials / total_trials
finite_artifact_rate = completed_trials_with_finite_daily_outputs / total_trials
pruned_trials 作为 pruned_trials 单独统计；除非 pruned trial 没有 finite validation artifact，否则不计入 failed_trials
```

## 8. P14 Aggregation

P14 的目标是把 promoted P12/P13 formal results 与既有 P7/P9 formal results 合并。

聚合前置：

- 只读 HPO per-model final outputs。
- 不读 parent best payload 当作额外模型行。
- 不重复读取 `baseline_comparison.csv` 与 `hpo_model_final_comparison.csv` 中的同一模型。
- 必须启用 formal manifest 和 availability-mask contract 过滤。

目标命令属于 target-after-implementation。执行前必须确认 `paper_aggregate` 支持 `--protocol-id`、`--data-cutoff-date`、`--require-formal-manifest`、`--require-availability-mask-contract`；若当前分支缺少任一参数，先实现 CLI 再执行。

目标命令：

```bash
.venv/bin/python -m src.experiments.paper_aggregate \
  --run-dir results/<p7_main_hpo_final_seed_*/> \
  --run-dir results/<p9_related_work_hpo_seed_*/> \
  --run-dir results/<p12_cage_eiie_formal_seed_*/> \
  --run-dir results/<p13_gt_rcpo_lite_formal_seed_*/> \
  --output-dir results/paper_tables/p14_new_model_final \

  --require-availability-mask-contract
```

`<promoted_primary_cage_model>` 由 validation-only promotion report 决定，不能硬编码成某个 CAGE 变体。

## 13. No-Go 条件

任一条件触发时，P12/P13 不得进入论文主表：

- 使用 final test ranking 选择 P12/P13 变体。
- smoke/pilot rows 标为 rankable。
- P12 输出预混合 target weights 且同时传 `rebalance_intensity=rho`。
- 模型 observation 使用 `pre_execution_drifted_weights`。
- rolling correlation 使用 full-sample 或 test-window 统计量。
- formal HPO 缺 `hpo_search_space_manifest.csv`。
- formal manifest 或 availability-mask contract 缺失。
- P14 聚合读取 parent best payload 导致重复模型行。
- risk metrics 中 Sharpe / MDD / CVaR 系统性 NaN，但论文声称 risk-adjusted improvement。

## 14. 论文写作边界

P12/P13 是 post-hoc extension，论文中必须写明：

```text
The P12/P13 models were introduced after observing the Core-13 v2 baseline results.
They reuse the same data, execution, cost, and evaluation protocol but constitute a registered model-extension addendum.
Promotion decisions are made on validation references only, and final-test results are used only for locked evaluation.
```

可主张：

- CAGE-EIIE 分离 allocation alpha 与 execution intensity control。
- P12 验证 EIIE candidate + cost-aware gate 是否改善 return-cost-risk trade-off。
- P13 验证 graph/transformer/risk-constrained policy 在 Core-13 小资产池上的边际价值。

不可主张：

- P12/P13 是完全独立的原始 confirmatory experiment。
- P12/P13 与 P7/P9 在未对齐 HPO budget 时可直接主表排名。
- 未通过 promotion gate 的 scout 模型可进入正式主表。
