## 1. 问题结论

当前低换手不是单纯的模型局部最优，而是由执行调度、gate 规则、HPO 目标和训练目标共同造成：

```text
monthly scheduler
+ hand-written rho scoring with scale mismatch
+ hold score = 0
+ HPO metric penalizes average_turnover directly
+ P12/P16 rho is not learned by policy gradient
= strategies converge to first-day build + long hold
```

关键证据：

```text
P16 formal:
  rho_gt0_count = 1 / 450
  realized_turnover_sum = 0.5
  forced_hold_reason:
    scheduler_blocked = 427
    model_chosen_hold = 22

P12 formal:
  each CAGE variant roughly only trades on first build / very few monthly dates.
```

## 2. 设计目标

### 2.1 必须修复

1. 不再让 RL / new-model gate 因量纲问题必然选择 hold。
2. 不再让 formal HPO 单纯偏好最低换手。
3. 明确区分 scheduler blocking 和 model-chosen hold。
4. 对 P12/P16 增加最小交易活跃度门禁，防止 buy-and-hold 化模型进入 formal 主表。
5. 让 P16 的 `rho` 成为可训练/可优化的策略输出，而不是完全由手写 `choose_rho()` 决定。

### 2.2 不做的事

```text
不取消交易成本。
不强迫每日调仓。
不允许用测试集选择换手阈值。
不把高换手本身当成成功。
不破坏 availability_mask / next_open / valuation-execution split。
```

---

## 3. 总体方案

分三层修：

```text
Layer A: Execution opportunity
  formal 主实验保留 monthly；
  新增 active-diagnostic / active-formal 分支支持 weekly 或 daily-with-gate；
  scheduler_allowed_rebalance 与 model_chosen_hold 分开统计。

Layer B: Gate scoring
  修正 choose_rho() 的量纲；
  增加 alpha threshold / opportunity cost of hold；
  增加 min active trade constraints。

Layer C: Training and HPO
  HPO objective 改为约束式或分段惩罚；
  P16 actor-critic 输出 rho / option；
  promotion gate 加交易活跃度和 active-risk 指标。
```

## 5. Layer A：执行机会与 scheduler 改造

### 5.1 保留现有 monthly formal，但新增 active protocol

现有 Core-13 v2 formal 默认：

```text
rebalance.mode = monthly
calendar_position = last_trading_day
```

这对传统定期再平衡合理，但对 gate / partial rebalance 模型会严重减少探索机会。

新增两个协议标签：

```text
execution_activity_protocol = monthly_gate
execution_activity_protocol = weekly_gate
execution_activity_protocol = daily_gate_with_cost_constraint
```

配置示例：

```yaml
rebalance:
  mode: daily
  threshold_turnover: 0.0
  threshold_weight_drift: 0.0

execution_activity:
  protocol: daily_gate_with_cost_constraint
  scheduler_blocks_model_actions: false
  activity_gate_enforced: true
  turnover_optimization_protocol_id: turnover_active_v1
  min_model_rebalance_hit_rate: 0.05
  max_model_rebalance_hit_rate: 0.60
```

上述 `execution_activity.*` 是目标态 schema。实现前必须扩展 `src/config.py`、`src/envs/backtest_engine.py` 与 `src/envs/rebalance_scheduler.py`；若 active formal 配置缺失 `execution_activity.protocol` 或 `scheduler_blocks_model_actions`，必须 fail fast，不能静默退回 legacy scheduler 行为。

解释：

```text
monthly_gate:
  论文主协议旧口径，可继续报告。

weekly_gate:
  给 gate 每周一次决策机会。

daily_gate_with_cost_constraint:
  scheduler 不主动阻止模型，只由 gate / cost / risk constraints 决定是否交易。
```

schema invariant：

```text
monthly_gate:
  scheduler_blocks_model_actions = true

weekly_gate:
  scheduler_blocks_model_actions = true
  # weekly_gate + scheduler_blocks_model_actions=false 非法，必须 fail fast。

daily_gate_with_cost_constraint:
  scheduler_blocks_model_actions = false
  # 每日给模型执行机会，成本/约束仍由 execution_core / constraint manager 控制。
```

`scheduler_blocks_model_actions: false` 是目标态配置，当前代码不会自动生效。实现必须同步修改：

```text
src/config.py
src/envs/backtest_engine.py
src/envs/rebalance_scheduler.py
```

新语义：

```text
scheduler_blocks_model_actions = true:
  现有行为，scheduler pre-check 可直接阻断执行。

scheduler_blocks_model_actions = false:
  strategy 先输出 raw model intent；
  BacktestEngine / PortfolioRebalanceEnv 再记录 scheduler outcome；
  active protocol 下 scheduler 不覆盖模型 gate，只保留诊断；
  availability、交易成本、约束与执行合法性仍由 execution_core / constraint manager 处理。
```

### 5.2 BacktestEngine 增加调仓机会诊断

在 `src/envs/backtest_engine.py` 的 daily diagnostics 增加：

```text
scheduler_pre_allowed

scheduler_final_allowed

raw_rebalance_intensity
raw_model_requested_rebalance

non_initial_trade_opportunity_count
model_rebalance_hit_rate

non_initial_turnover_sum

```

定义：

```text
raw_rho = strategy action_info["raw_rho"] before scheduler masking
raw_rebalance_intensity = raw_rho
raw_model_requested_rebalance = raw_rho > 0
raw_action = int(raw_model_requested_rebalance)

scheduler_pre_allowed, scheduler_post_allowed =
  RebalanceScheduler.evaluate_pre_post_no_mutation(...)

scheduler_pre_blocked =
  raw_model_requested_rebalance
  and scheduler_pre_allowed == false

scheduler_post_blocked =
  raw_model_requested_rebalance
  and scheduler_pre_allowed == true
  and scheduler_post_allowed == false

scheduler_final_allowed = scheduler_pre_allowed and scheduler_post_allowed
scheduler_final_blocked =
  raw_model_requested_rebalance
  and scheduler_final_allowed == false

if scheduler_blocks_model_actions == true:
  execution_gate_allowed = scheduler_final_allowed
else:
  execution_gate_allowed = true

trade_opportunity_count = count(execution_gate_allowed or first_trade)
non_initial_trade_opportunity_count = count(execution_gate_allowed excluding first_trade)
final_action = int(raw_model_requested_rebalance and execution_gate_allowed)
final_rho = raw_rho if final_action == 1 else 0.0
execution_action.rebalance_intensity = final_rho

scheduler_blocked_rebalance = scheduler_final_blocked
execution_scheduler_blocked =
  raw_model_requested_rebalance
  and final_action == 0
  and scheduler_blocks_model_actions == true
  and scheduler_final_allowed == false
model_chosen_hold =
  not first_trade
  and execution_gate_allowed == true
  and raw_rho == 0
non_initial_rebalance_count = count(final_action == 1 excluding first_trade)
active_weight_change_l1 = sum(abs(executed_weights_t - pre_execution_weights_t))
model_rebalance_hit_rate = non_initial_rebalance_count / max(1, non_initial_trade_opportunity_count)
non_initial_turnover_per_opportunity =
  non_initial_turnover_sum / max(1, non_initial_trade_opportunity_count)
```

策略侧只能新增 raw intent：

```text
action_info["raw_rho"]
action_info["raw_rebalance_intensity"]
action_info["raw_model_requested_rebalance"]
action_info["raw_gate_action_index"]
```

engine/env 追加 final 与 scheduler 字段：

```text
action_info["final_rho"]
action_info["final_rebalance_intensity"]
action_info["scheduler_pre_blocked"]
action_info["scheduler_post_blocked"]
action_info["scheduler_final_blocked"]
action_info["execution_scheduler_blocked"]
```

P12/P16 不得在 `scheduler_allowed == false` 时直接把 raw intent 清零后再返回，否则无法区分“模型想交易但 scheduler 阻断”和“模型主动 hold”。

`raw_rho` 只能来自模型或规则 scorer 的原始意图；`final_rho` 只能由 BacktestEngine / PortfolioRebalanceEnv / execution activity protocol 统一生成。策略层不得用 scheduler mask 改写 `raw_rho`。

### 5.3 验收

```text
monthly formal:
  scheduler_allowed_rebalance 可低，只披露，不设 active activity 门槛。

weekly_gate:
  trade_opportunity_count / n_steps >= 0.15 或按实际周数校验。
  model_rebalance_hit_rate 使用 non_initial_trade_opportunity_count 作分母。

daily_gate_with_cost_constraint:
  trade_opportunity_count / n_steps >= 0.80
  model_rebalance_hit_rate 不得低于 configured min threshold
  scheduler_blocked_rebalance 可作为诊断披露；
  execution_scheduler_blocked 必须为 0，除非显式配置 scheduler_blocks_model_actions=true。
```

## 6. Layer B：修正 choose_rho()

### 6.1 当前问题

当前 `src/baselines/cage_common.py::choose_rho()`：

```text
score(rho) =
  rho * expected_return
  - lambda_turnover * rho * estimated_turnover
  - lambda_cost * rho * estimated_cost
  - ...

score(0) = 0
```

量纲问题：

```text
expected_return: 0.0001 ~ 0.002
estimated_turnover: 0.01 ~ 0.10
lambda_turnover: 1 ~ 5
```

结果：

```text
lambda_turnover * estimated_turnover >> expected_return
score(rho > 0) almost always < 0
=> hold
```

### 6.2 新增归一化 gate scorer

新增：

```python
score_rho_normalized(
    rho_values,
    expected_alpha,
    estimated_turnover,
    estimated_cost,
    cvar_loss_5,
    drawdown,
    scale_config,
)
```

新公式：

```text
alpha_score = expected_alpha_horizon / alpha_scale
turnover_excess_score = max(0, rho * estimated_turnover - turnover_budget_per_trade) / turnover_scale
cost_excess_score = max(0, rho * estimated_cost - cost_budget_per_trade) / cost_scale
cvar_score = max(0, cvar_loss_5 - cvar_budget) / cvar_scale
drawdown_score = max(0, drawdown - drawdown_budget) / drawdown_scale

score(rho) =
  rho * alpha_score
  - lambda_turnover * turnover_excess_score
  - lambda_cost * cost_excess_score
  - lambda_cvar * rho * cvar_score
  - lambda_drawdown * rho * drawdown_score
  + hold_opportunity_penalty * I[rho == 0 and alpha_score > alpha_activation_threshold]
```

说明：

```text
expected_alpha_horizon 必须与当前调仓机会对应的预测期一致。
monthly_gate 用月度持有期 alpha；daily_gate 用下一交易日 alpha。
turnover/cost 默认只惩罚超预算部分，不再惩罚预算内必要调仓。
```

`expected_alpha_horizon` 统一由以下纯函数生成：

```text
compute_expected_alpha_horizon(
  decision_ts,
  activity_protocol,
  candidate_weights,
  current_weights,
  mu_1d_decision_visible,
  horizon_config,
) -> expected_alpha_horizon
```

定义：

```text
expected_alpha_1d =
  dot(candidate_weights - current_weights, mu_1d_decision_visible)

horizon_days:
  daily_gate_with_cost_constraint = 1
  weekly_gate = 5
  monthly_gate = 21

expected_alpha_horizon =
  clip(horizon_days * expected_alpha_1d, -alpha_cap, alpha_cap)
```

约束：

```text
mu_1d_decision_visible 只能来自 decision date 可见的模型预测、训练期统计或滚动窗口估计。
不得使用 validation/test realized future return、全样本均值、post-hoc regime label 或 execution-only 字段。
若模型提供显式 multi-step forecast，可替代线性 horizon scaling，但必须记录 forecast_horizon_days 与 estimator_id。
```

默认 scale：

```yaml
gate_scoring:
  mode: normalized
  alpha_scale: 0.001
  turnover_scale: 0.05
  cost_scale: 0.001
  cvar_scale: 0.01
  drawdown_scale: 0.05
  alpha_activation_threshold: 0.25
  hold_opportunity_penalty: -0.20
  turnover_budget_per_trade: 0.05
  cost_budget_per_trade: 0.001
  lambda_turnover: 0.20
  lambda_cost: 0.20
  lambda_cvar: 0.20
  lambda_drawdown: 0.20
```

说明：

```text
hold_opportunity_penalty 不是强迫交易。
它只在 expected alpha 明显为正时降低无脑 hold 的吸引力。
normalized scorer 不复用 legacy_raw 的 lambda 搜索空间。
默认 lambda 降到 0.05~0.50 区间，并在 HPO search-space manifest 中单独披露。
```

### 6.3 保留旧 scorer 用于对照

```yaml

  mode: legacy_raw
```

正式新实验必须记录：

```text

cost_scale
hold_opportunity_penalty
```


```text
rho_logits
rho_probs
rho_action_index

```

### 7.3 Agent 训练新增 rho loss

修改 `src/agents/constrained_actor_critic_agent.py`：

```text
policy objective =
  after_cost_return
  - risk penalties
  - constraint penalties

rho policy:
  mode = straight_through_gumbel_softmax_v1
  executed_weights = (1 - rho) * current_weights + rho * candidate_weights
  portfolio_return = executed_weights dot future_returns
  turnover = 0.5 * abs(executed_weights - current_weights).sum()
```

训练期：

```text
rho_probs = softmax(rho_logits)
rho_one_hot_st = gumbel_softmax(rho_logits, tau, hard=True)
rho_train = sum(rho_one_hot_st * rho_values)
```

温度退火协议：

```yaml
ra_gt_rcpo:
  rho_temperature:
    tau_start: 1.0
    tau_end: 0.20
    tau_decay_steps: 2048
    eval_mode: argmax
    min_entropy_threshold: 0.05
```

定义：

```text
tau(step) = max(tau_end, tau_start * exp(-step / tau_decay_steps))
formal 若 validation rho_entropy < min_entropy_threshold 且 model_rebalance_hit_rate 不达标，则返回 failed_rho_policy_collapsed。
```

评估期：

```text
deterministic:
  rho = rho_values[argmax(rho_logits)]

stochastic:
  rho ~ Categorical(rho_probs)
```

`differentiable_expected_rho` 只能作为 diagnostic ablation，不能作为 P16 active formal 主实现。否则 train 用连续期望、eval 用离散 argmax，会产生训练/评估口径不一致。

必须记录：

```text
rho_logits
rho_probs
rho_entropy
rho_expected
rho_action_index
rho_policy_mode
rho_temperature
```

注意：

```text
训练里的 executed_weights 只用于 loss。
正式回测必须交给 BacktestEngine / execution_core 执行；训练 env 必须使用同一 raw/final action 合同。
```

### 7.4 策略执行路径

`RiskAwareGTRCPOStrategy.compute_target_weights()`：

```text
if config.ra_gt_rcpo.rho_policy = learned:
  raw_rho = agent_output.raw_rho
else:
  raw_rho = score_rho_normalized(...)

raw_strategy_action = PortfolioAction(
  target_weights = candidate,
  rebalance_action = int(raw_rho > 0),
  rebalance_intensity = raw_rho,  # raw intent only; not passed to execution_core directly
  action_info = {
    raw_rho,
    raw_rebalance_intensity: raw_rho,
    raw_model_requested_rebalance: raw_rho > 0,
  },
)
return raw_strategy_action
```

BacktestEngine / PortfolioRebalanceEnv 再统一构造 `execution_action`：

```text
scheduler_pre_allowed, scheduler_post_allowed =
  scheduler.evaluate_pre_post_no_mutation(...)
scheduler_final_allowed = scheduler_pre_allowed and scheduler_post_allowed

if scheduler_blocks_model_actions:
  execution_gate_allowed = scheduler_final_allowed
else:
  execution_gate_allowed = true

final_action = raw_model_requested_rebalance and execution_gate_allowed
final_rho = raw_rho if final_action else 0.0

execution_action = replace(
  raw_strategy_action,
  rebalance_action = int(final_action),
  rebalance_intensity = final_rho,
)
execution_core receives execution_action only
sidecar records raw_rho / final_rho / scheduler diagnostics
```

`evaluate_pre_post_no_mutation()` 必须是纯诊断评估，不得调用会推进 scheduler 状态的 `should_rebalance()`。若现有 scheduler 没有纯函数式接口，需在 `src/envs/rebalance_scheduler.py` 新增该接口，再由 BacktestEngine 与 PortfolioRebalanceEnv 共用。

scheduler 状态提交点：

```text
commit_scheduler_decision(
  decision_ts,
  scheduler_pre_allowed,
  scheduler_post_allowed,
  scheduler_final_allowed,
  raw_model_requested_rebalance,
  final_action,
  execution_accepted,
)
```

规则：

```text
evaluate_pre_post_no_mutation() 不得修改 _last_allowed_date / _has_rebalanced。
commit_scheduler_decision() 是唯一允许修改 scheduler 状态的入口。
scheduler_blocks_model_actions=true 时，commit 后状态必须与 legacy should_rebalance() 路径一致。
scheduler_blocks_model_actions=false 只允许 daily_gate_with_cost_constraint；weekly/monthly 禁止该组合。
```

语义矩阵：

```text
scheduler_blocks_model_actions = true:
scheduler_final_allowed = scheduler_pre_allowed and scheduler_post_allowed
final_action = raw_model_requested_rebalance and scheduler_final_allowed
final_rho = raw_rho if final_action else 0.0

scheduler_blocks_model_actions = false:
final_action = raw_model_requested_rebalance
final_rho = raw_rho if final_action else 0.0
scheduler_* 字段只进入诊断，不阻断执行。
```

### 7.6 P16 最低训练预算

P16 learned rho active formal 不得沿用 8 次 gradient update 的 smoke 预算。最低正式预算：

```yaml
baselines:
  native_rl:
    max_train_steps: 2048
    max_validation_steps: 512
    max_gradient_updates_per_epoch: 64
training:
  epochs: 8
  early_stopping_patience: 3
  min_gradient_updates_for_formal: 128
  min_env_steps_for_formal: 2048
```

formal gate：

```text
gradient_updates >= min_gradient_updates_for_formal
env_steps >= min_env_steps_for_formal
validation_metric 至少一次 finite improvement
rho_entropy 不得在前 10% updates 内坍缩到 < 0.05
```

不满足时：

```text
status = failed_insufficient_training_budget
rankable_in_unified_table = false
```

## 8. HPO objective 改造

### 8.1 当前问题

`src/experiments/pipeline.py::objective_metric()`：

```text
validation_sharpe_minus_drawdown_turnover_penalty =
  sharpe - max_drawdown - average_turnover
```

这会持续偏好低换手。

### 8.2 新增约束式 objective

新增 objective：

```text
validation_return_risk_cost_constrained
```

公式：

```text
base = sharpe - max_drawdown_loss - cvar_loss_5

activity_underuse_penalty =
  hit_rate_underuse_penalty * max(0, min_model_rebalance_hit_rate - model_rebalance_hit_rate)
  + turnover_underuse_penalty * max(0, min_non_initial_turnover_per_opportunity - non_initial_turnover_per_opportunity)

turnover_overuse_penalty_value =
  turnover_overuse_penalty * max(0, average_turnover - max_average_turnover)

cost_penalty =
  cost_over_budget_penalty * max(0, total_transaction_cost - cost_budget)

objective = base - activity_underuse_penalty - turnover_overuse_penalty_value - cost_penalty
```

风险指标来源优先级：

```text
1. validation daily_returns 重新计算 sharpe / max_drawdown_loss / cvar_loss_5。
2. 只有当 metrics 字段明确来自同一 validation split 时，才可复用现有 metrics。
3. 缺失、NaN、inf 或 split provenance 不一致时，trial 失败为 failed_metric_unavailable。
```

配置：

```yaml
hpo:
  metric: validation_return_risk_cost_constrained
  objective: validation_return_risk_cost_constrained
  activity_constraints:
    enabled: true
    scope_baseline_families: [new_model_extension, platform_native_rl]
    scope_activity_protocols: [weekly_gate, daily_gate_with_cost_constraint]
    min_model_rebalance_hit_rate: 0.05
    max_model_rebalance_hit_rate: 0.60
    min_non_initial_turnover_per_opportunity: 0.002
    max_average_turnover: 0.030
    hit_rate_underuse_penalty: 5.0
    turnover_underuse_penalty: 5.0
    turnover_overuse_penalty: 2.0
    cost_budget: 0.010
    cost_over_budget_penalty: 10.0
```

### 8.3 新增 hard fail 规则

HPO hard fail 只允许作用于 active protocol 的可训练模型：

```text
activity_constraints.enabled = true
activity_gate_enforced = true
baseline_family in {new_model_extension, platform_native_rl}
activity_protocol in {weekly_gate, daily_gate_with_cost_constraint}
```

不得作用于：

```text
traditional deterministic baselines
monthly_gate formal
fixed buy-and-hold diagnostic
```

满足作用范围后，HPO trial 若满足以下条件，标记低活跃失败：

```text
model_rebalance_hit_rate < min_model_rebalance_hit_rate
or non_initial_turnover_per_opportunity < min_non_initial_turnover_per_opportunity
```

状态：

```text
failed_low_trade_activity
```

该状态不得进入 active protocol 的 best trial。

统一收益风险主表仍必须保留低活跃模型的诊断结果，不能用 hard fail 隐藏“低频更优或主动交易失败”的结论。

## 14. 成功标准

优化后模型不得只满足低成本，还必须满足活跃度：

```text
daily_gate_with_cost_constraint:
  model_rebalance_hit_rate >= 5%
  non_initial_turnover_per_opportunity >= 0.002

weekly_gate:
  model_rebalance_hit_rate >= 15%
  non_initial_turnover_per_opportunity >= 0.010

monthly_gate:
  只披露 activity，不设 hard activity 成功门槛。

test_used_for_model_selection = false
daily_returns/nav finite
availability_mask_contract_passed = true
```

性能成功标准采用 Pareto：

```text
相对 CAGE/EIIE:
  cumulative_return 不显著恶化
  或 Sharpe / MDD / CVaR 至少两个改善
  且换手和成本不爆炸
```

若 active model 仍不优：

```text
说明 Core-13 v2 更适合低频配置；
论文不能继续声称复杂 RL gate 有主动交易优势。
```

## 15. 风险

```text
R1: daily_gate 可能提升噪声交易。
  缓解：max_average_turnover / cost budget hard constraint。

R2: min activity gate 可能强迫无意义交易。
  缓解：只作为 promotion requirement，不直接进入 reward；并设置成本上限。

R3: learned rho 增加训练不稳定。
  缓解：formal 使用 straight-through Gumbel-Softmax；differentiable expected rho 仅做 diagnostic。

R4: 新旧 formal 不可直接混排。
  缓解：新增 activity_protocol 与 turnover_optimization_protocol_id 字段，主表分 monthly formal 与 active formal。
```
