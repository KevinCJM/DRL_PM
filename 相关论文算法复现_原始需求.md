## 5. Baseline 1：PPO-DQN Hierarchical Reimplementation

baseline id：

```text
ppo_dqn_hierarchical_reimplementation
```

需求定位：

```text
该 baseline 代表 PPO-DQN hybrid / hierarchical portfolio allocation 路线。
该 baseline 是相近论文算法复现，不是本项目 full_dqn_gated_multitask_cnn_ppo 的 gate 消融。
该 baseline 中的 DQN 不得默认解释为“本文 binary rebalance gate”。
```

必须满足：

```text
1. PPO 组件负责连续组合权重或候选权重生成。
2. 首版正式验收只接受 dqn_role=high_level_action_selector。
3. DQN 负责选择 high-level hierarchy action，用于调节 PPO 候选权重或 fallback 权重。
4. high-level action set 若不是原论文明确给出的精确定义，必须标记 platform_adapted。
5. 必须清楚记录 hierarchy_action 与 hierarchy_action_name。
6. high-level action 不得与本项目主模型 hold/rebalance gate 混用。
7. PPO actor loss 必须只归因于实际执行或可正确计算 log_prob 的候选权重。
8. DQN transition 必须来自真实环境交互，不得伪造 next_state。
9. 训练完成后必须保存 best / last checkpoint，并在 final test 前加载 best checkpoint。
```

延期范围：

```text
auxiliary_value_support 可作为后续可选变体，但不属于首版正式验收。
若后续实现 auxiliary_value_support，必须标记：
  dqn_role=auxiliary_value_support
  status=deferred_variant 或 needs_paper_confirmation
且不得进入首版 main comparable table。
```

最低元数据：

```text
model_name = ppo_dqn_hierarchical_reimplementation
baseline_family = native_rl_reimplementation
training_algorithm = ppo_dqn_hierarchical_reimplementation
clean_room_reimplementation = true
algorithm_fidelity = exact_from_paper | platform_adapted | assumption
dqn_role = high_level_action_selector
platform_native_rl_training = true
proxy_training = false
rankable_in_unified_table = true
```

验收要求：

```text
1. 能在平台 train split 上完成真实 RL 训练。
2. 能在 validation split 上产生有限 validation_metric。
3. 能在 test split 上通过 BacktestEngine 生成完整日度产物。
4. training_history 至少记录 epoch、env_steps、gradient_updates、train_reward、validation_metric、loss、status。
5. daily_rebalance 或诊断产物中能追溯 hierarchy action 分布。
6. comparison 表中能看到 algorithm_fidelity、dqn_role、hierarchy_action 分布摘要。
```

baseline id：

```text
hybrid_dqn_optimizer_reimplementation
```

```text

```

```text

```

正式输出粒度：

```text
hybrid_dqn_optimizer_reimplementation 是算法族 baseline id。

hybrid_dqn_optimizer_equal_weight

hybrid_dqn_optimizer_minimum_variance
hybrid_dqn_optimizer_sharpe_maximization

comparison 表至少应有五个 optimizer 子模型行。
```

必须满足：

```text
1. DQN 输出每个资产的离散信号，例如 exclude / neutral / include。

5. 优化器失败时必须 fallback 到可解释、可记录的等权方案。
6. DQN 训练使用组合级真实环境 reward，不得伪造资产级反事实收益。
7. 若使用 factorized Q 或 per-asset Q approximation，必须记录 platform_adapted_approximation=true。
8. final target_weights 必须通过平台约束和 BacktestEngine 执行。
```

最低元数据：



## 8. HPO 与公平比较要求

正式 rankable 版本必须能参加平台 equal-budget HPO。smoke / pilot 允许使用短预算，但只能进入 diagnostic table，不能进入 main comparable table。

公平预算至少包括：

```text
hpo_trials
env_steps
gradient_updates
random_seeds
validation_metric
selection_rule
train/validation/test split
execution protocol
cost protocol
constraint protocol
```

主表资格：

```text
1. 只有 rankable_in_unified_table=true 的平台内 native reimplementation 才能进入主排名。
2. 若某 baseline 使用非同预算、非统一执行、非统一成本或 external result，只能进入 appendix / diagnostics。
3. HPO final comparison 必须保留 per-model final daily outputs 与 comparison metadata。
4. paired statistics 必须使用同一 test period 的 daily net returns。
```

正式 HPO 与 seed 协议：

```text
1. 正式 main comparable table 采用每个 seed 独立 HPO。
2. 每个 seed 内，每个 trainable model 或 optimizer 子模型获得相同 hpo_trials、env_steps、gradient_updates、validation_metric 与 selection_rule。
3. 每个 seed 的 final test 使用该 seed validation 选出的 best checkpoint。
4. 正式表按 seed 聚合均值、标准差、最小值、最大值和中位数。
5. 禁止使用“单次 HPO 选 best config 后再 5-seed retrain”作为默认正式口径，除非单独标记为 alternative_protocol。
6. pilot HPO + formal fixed-budget final run 只能作为诊断或工程预演，不能替代正式同预算 HPO 主表。
```

Hybrid DQN optimizer 的 HPO 预算：

```text
正式 rankable 表中，五个 optimizer 子模型按独立 trainable model 计预算。
即每个 optimizer 子模型分别训练 DQN signal，并分别进行 validation selection 与 final test。
共享一个 DQN 后对五个 optimizer 分别 final test 的方案只能作为 diagnostic_shared_dqn，不得进入正式主可比表。
```

长实验执行说明：

```text
1. 每 seed 独立 HPO 是正式论文主表口径，不因耗时降低为共享 HPO 或共享 DQN。
2. 编排任务应标记 long_running=true，并支持 checkpoint、resume、progress logging 或定时检查。
3. smoke / pilot 可使用短预算验证链路，但必须标记为 diagnostic，不得替代正式 HPO 结果。
```

## 12. 完成定义

该需求完成的定义：

```text
1. DRL_PM 平台中存在两个新增 clean-room native reimplementation baseline。
2. 两个 baseline 均能训练、验证、保存 checkpoint、加载 best checkpoint、统一回测。
3. PPO-DQN 首版以 high_level_action_selector 口径完成验收。
4. Hybrid DQN optimizer 以五个 optimizer 子模型口径完成验收。
5. 两个 baseline 均能参加同预算 HPO；未参加正式同预算 HPO 的结果必须明确标记为非正式诊断。
6. 所有输出均通过平台 logger / aggregation / paper artifact bundle 闭环。
7. 论文能将两个 baseline 作为最相近 hybrid 相关工作复现实验进行对比。
8. 实验解释清楚区分：
   - 原论文结果；
   - 平台内 clean-room reimplementation 结果；
   - 本项目 DQN-Gated PPO 主模型结果。
```
