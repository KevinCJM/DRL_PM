## 2. 固定正式协议

### 2.1 主协议记录，不是 ConfigLoader YAML

本节是 protocol / run manifest 口径说明，**不能直接复制进 `configs/paper/*.yaml`**。当前 ConfigLoader 不支持 `protocol_id`、`asset_universe_id`、`data.data_mode`、`data.return_source`、`data.execution_price_source` 这类字段；正式 YAML 只能使用 `src/config.py` 当前 schema。

```yaml
protocol_id: core13_v2_full_reset_20260522
asset_universe_id: core13_v2
data_cutoff_date: 2026-05-20
data_contract:
  data_mode: availability_mask
  strict_common_history_mode: false
  return_source: adj_nav
  valuation_source: adj_nav
  reward_return_source: adj_nav
  metrics_return_source: adj_nav
  execution_price_source: ohlcv
feature_matrix:
  input_matrix_id: M6
feature_reduction:
  pca:
    enabled: true
    explained_variance: 0.95
execution_model:
  strict_no_lookahead_execution: true
  execution_price: next_open
portfolio:
  initial_capital_currency: 100000000.0
  currency: CNY
cost_model:
  proportional_cost: 0.001
  slippage: 0.0005
  market_impact_enabled: true
  market_impact_coef: 0.10
rebalance:
  mode: monthly
reward:
  mode: A2_net_log_return_after_cost
```

正式配置映射：

```text
protocol_id / asset_universe_id -> run_manifest 或 sidecar manifest，不写入 ConfigLoader YAML。
data_contract.data_mode -> run_manifest 或 sidecar manifest；当前 YAML 只写 data.strict_common_history_mode=false。
data_contract.return_source / valuation_source / reward_return_source / metrics_return_source / execution_price_source -> data_governance 或 sidecar manifest。
cost_model.* -> 当前 ConfigLoader 的 cost_model 节点，不使用 transaction_cost 节点。
```

### 2.2 不是笛卡尔积实验

完整实验不等于全参数笛卡尔积。

```text
主协议参数：预注册固定。
模型超参数：equal-budget HPO 搜索。
消融参数：一次只动一个开关。
稳健性参数：分层诊断，不混入主排名。
```

禁止执行：

```text
input_matrix × PCA × cost × rebalance × reward × encoder × seed × model
```

的完整笛卡尔积。

  "data_mode": "availability_mask",

strict_common_retained_start/end：严格共同历史筛选后保留日期区间。

不得把 strict-common retained start 当成 availability-mask formal 的数据开始日期。

### 5.3 Core-13 v2 data mode 裁决

本版正式协议裁决为：

```yaml
data:
  strict_common_history_mode: false
run_manifest:
  data_mode: availability_mask
```

原因：当前 Core-13 配置和 manifest 已走 availability-mask 分支；strict common history retention 约低于 0.90，不能直接声称 strict formal。

availability-mask formal 必须显式记录：

```yaml
run_manifest:
  data_mode: availability_mask
  availability_mask_contract_passed: true
  min_available_assets_per_date: 2
  unavailable_asset_weight_abs_max: 0.0
  daily_returns_finite: true
  daily_nav_finite: true
  frozen_or_imputed_valuation_count: 0
```

且不得与 strict common history 结果混合排名。

availability-mask contract 硬门禁：

```text
1. 每个 decision_date / execution_date 至少有 2 个可交易资产。
2. 不可用资产在 daily_weights 中的绝对权重最大值必须等于 0。
3. daily_returns.net_return 与 nav 必须全为 finite。
4. availability_reason_counts 必须可追溯到 not_listed_yet / missing_quote / missing_return / suspended / delisted。
5. 若存在 frozen 或 imputed valuation，必须记录 frozen_or_imputed_valuation_count 与字段来源；正式主表默认要求为 0。
6. 决策日 mask 与执行日 tradeable mask 的来源必须记录，不能只写 data_mode 标签。
```

若上述字段或校验逻辑尚未实现，则 `availability_mask` run 只能是 diagnostic，不能进入 formal Table A / G / H。

若后续改回 `strict_common_history`，必须重新生成 Core-13 schema / MetricsFactory / manifest，并满足：

```text
calendar_loss_retention_ratio >= 0.90
data_manifest.all_assets_available_each_date = true
```
