# 相关论文算法复现_需求澄清

- 目标闭环
  - 输入：`相关论文算法复现_原始需求.md` [原始需求文档] 与 `相关论文算法复现_人机交互澄清记录.md` [人机澄清事实缓存]。
  - 规则：在 DRL_PM 统一实验平台内新增 clean-room native reimplementation baseline；所有可排名结果必须遵守统一数据、切分、next-open 执行、成本、约束、HPO、统计检验与 paper aggregation 协议。
  - 输出：两个相关论文算法复现能力进入平台：`ppo_dqn_hierarchical_reimplementation` [PPO-DQN 层级复现 baseline] 与 `hybrid_dqn_optimizer_reimplementation` [DQN 信号加组合优化器算法族标识]。
  - 异常兜底：任何不能完成训练、validation best checkpoint、统一回测或 finite validation metric 的运行不得进入正式主表。

- baseline 身份与配置入口
  - 输入：`src/experiments/registry.py::DEEP_BASELINE_CLASSES` [baseline 名称到策略类的注册表]、`baselines.native_rl.enabled_models` [native baseline 配置入口]、`hpo.trainable_models` [HPO 可训练模型列表]。
  - 规则：
    - 正式可排名实验只使用六个可训练条目：
      - `ppo_dqn_hierarchical_reimplementation` [PPO-DQN 层级复现 baseline]
      - `hybrid_dqn_optimizer_equal_weight` [DQN 信号 + 等权优化器子模型]
      - `hybrid_dqn_optimizer_markowitz_mean_variance` [DQN 信号 + 均值方差优化器子模型]
      - `hybrid_dqn_optimizer_minimum_variance` [DQN 信号 + 最小方差优化器子模型]
      - `hybrid_dqn_optimizer_sharpe_maximization` [DQN 信号 + 最大夏普优化器子模型]
      - `hybrid_dqn_optimizer_risk_parity` [DQN 信号 + 风险平价优化器子模型]
    - `hybrid_dqn_optimizer_reimplementation` [DQN 信号加组合优化器算法族标识] 只作为算法族标识与 orchestration alias；配置中出现时展开为五个 `hybrid_dqn_optimizer_*` [optimizer 子模型] child run。
    - `hybrid_dqn_optimizer_reimplementation` [DQN 信号加组合优化器算法族标识] 不作为 HPO budget 单元，不生成 rankable 主表行。
  - 输出：正式主表最多产生一个 PPO-DQN 行和五个 hybrid optimizer 子模型行。
  - 异常兜底：如果算法族 alias 无法完整展开为五个 child run，正式 / HPO / rankable run 标记 `failed` [失败状态]。

- `ppo_dqn_hierarchical_reimplementation` [PPO-DQN 层级复现 baseline]
  - 输入：
    - `PPO candidate_weights` [PPO 生成的候选组合权重]
    - `portfolio_state.current_weights` [当前持仓权重]
    - `DecisionMarketState.available_mask_at_decision` [决策时可用资产掩码]
    - `BacktestEngine` [统一回测执行器] validation split 与 test split
  - 规则：
    - `dqn_role` [DQN 角色] 固定为 `high_level_action_selector`。
    - `algorithm_fidelity` [算法忠实度] 标记为 `platform_adapted`。
    - `hierarchy_action` [PPO-DQN 高层动作编码] 首版固定五动作：
      - `0 use_ppo_candidate` [直接使用 PPO 候选权重]
      - `1 blend_current_ppo_25` [25% PPO 候选权重 + 75% 当前权重]
      - `2 blend_current_ppo_50` [50% PPO 候选权重 + 50% 当前权重]
      - `3 blend_current_ppo_75` [75% PPO 候选权重 + 25% 当前权重]
      - `4 fallback_equal_weight` [可用资产等权 fallback]
    - `candidate_weights` [PPO 候选权重] 与 `current_weights` [当前权重] 先按 `available_mask_at_decision` [决策时可用资产掩码] 做 mask + normalize。
    - blend 动作为 `(1-rho) * current_weights + rho * candidate_weights` [当前权重与 PPO 候选权重加权融合]。
    - 最终 `PortfolioAction.target_weights` [回测执行目标权重] 必须再经过 `ConstraintManager` [组合约束管理器] 与 `BacktestEngine` [统一回测执行器]。
    - `PPO actor loss` [PPO actor 损失] 归因规则：
      - `use_ppo_candidate` [直接使用 PPO 候选权重]：`ppo_actor_update_mask=1`，`ppo_attribution_weight=1.0`。
      - `blend_current_ppo_25` [25% PPO 候选权重]：`ppo_actor_update_mask=1`，`ppo_attribution_weight=0.25`，actor loss 使用 `PPO candidate log_prob * rho` [PPO 候选权重 log_prob 乘归因权重]。
      - `blend_current_ppo_50` [50% PPO 候选权重]：`ppo_actor_update_mask=1`，`ppo_attribution_weight=0.50`，actor loss 使用 `PPO candidate log_prob * rho` [PPO 候选权重 log_prob 乘归因权重]。
      - `blend_current_ppo_75` [75% PPO 候选权重]：`ppo_actor_update_mask=1`，`ppo_attribution_weight=0.75`，actor loss 使用 `PPO candidate log_prob * rho` [PPO 候选权重 log_prob 乘归因权重]。
      - `fallback_equal_weight` [可用资产等权 fallback]：`ppo_actor_update_mask=0`，`ppo_attribution_weight=0.0`，critic 仍训练，DQN 仍按真实 transition 更新。
    - `platform_adapted_surrogate=true` [平台适配 surrogate 标记] 必须记录在 blend 动作逐日诊断、训练摘要与 baseline 对比摘要中；blend actor loss 不是 `blended final weights` [融合后最终权重] 的精确 log_prob。
  - 输出：
    - `training_result["status"]` [训练状态] 为 `completed` 时，必须同时满足 `rl_training=true`、`platform_native_rl_training=true`、`proxy_training=false`、`external_original_implementation=false`、`rankable_in_unified_table=true`、`self.is_fitted=true`。
    - `baseline_daily_diagnostics.csv` [baseline 逐日 sidecar 诊断产物] 记录 `hierarchy_action` [高层动作编码]、`hierarchy_action_name` [高层动作名称]、`ppo_actor_update_mask` [PPO actor 更新掩码]、`ppo_attribution_weight` [PPO 收益归因权重]、`platform_adapted_surrogate` [平台适配 surrogate 标记]。
    - `baseline_training_summary.csv` [baseline 训练摘要表] 记录 `platform_adapted_surrogate` [平台适配 surrogate 标记]。
    - `baseline_comparison.csv` [baseline 对比表] 记录 `algorithm_fidelity` [算法忠实度]、`dqn_role` [DQN 角色]、`hierarchy_action` [高层动作] 分布摘要、`platform_adapted_surrogate` [平台适配 surrogate 标记]。
  - 异常兜底：
    - 无法产生有效 `PortfolioAction.target_weights` [回测执行目标权重] 时，训练或 trial 标记 `failed_no_valid_action`。
    - 无 finite `validation_penalized_sharpe` [验证集惩罚夏普业务指标] 时，标记 `failed_no_finite_validation_metric`。
    - 缺失或加载失败 best checkpoint 时，不得进入 final test。

- `hybrid_dqn_optimizer_*` [DQN 信号加组合优化器子模型]
  - 输入：
    - 每资产三动作 Q 值：`exclude` [排除]、`neutral` [中性]、`include` [纳入]
    - `DecisionMarketState.available_mask_at_decision` [决策时可用资产掩码]
    - `DecisionMarketState.log_return_window` [决策时可见收益历史窗口]
    - `BacktestEngine` [统一回测执行器] validation split 与 test split
  - 规则：
    - DQN 动作编码固定为：`0 exclude` [排除]、`1 neutral` [中性]、`2 include` [纳入]。
    - 不可用资产强制 `exclude` [排除]。
    - 默认只有 `include` [纳入] 资产进入优化器候选池。
    - `neutral` [中性] 不直接进入候选池；当 `include` [纳入] 资产不足时，可从所有 `非 include 且可用资产` [未被 DQN 判为 include 但仍可交易的资产] 中补足 top-k。
    - 补足排序分数固定为 `q_include - max(q_neutral, q_exclude)` [include 优势分数]。
    - `selected_asset_count` [原始 include 资产数量] 统计补足前的 include 数量。
    - `optimizer_asset_count` [优化器候选资产数量] 统计补足后的最终候选资产数量。
    - `min_selected_assets` [候选资产最低数量] 固定为 `2`。
    - 五个 optimizer 子模型均为独立可训练模型；正式 rankable 表中每个子模型分别训练 DQN signal、分别 validation selection、分别 final test。
    - 正式 paper config 必须显式 pin optimizer 参数：
      - `lookback_window=252` [历史窗口长度]
      - `min_observations=60` [最小历史样本数]
      - `covariance_shrinkage=0.1` [协方差收缩系数]
      - `risk_free_rate=0.0` [无风险利率]
      - `lambda_risk=1.0` [风险厌恶系数]
      - `markowitz optimizer_maxiter=200` [Markowitz 优化最大迭代次数]
      - `risk_parity optimizer_maxiter=300` [风险平价优化最大迭代次数]
    - `hybrid_dqn_optimizer_equal_weight` [DQN 信号 + 等权优化器子模型] 只使用 DQN 候选池，不使用协方差参数。
  - 输出：
    - 五个 child run 均产生独立 `paper_model_id` [论文模型标识]、`optimizer_name` [优化器名称]、`training_summary` [训练摘要]、`comparison row` [对比表行] 与 daily outputs。
    - `baseline_daily_diagnostics.csv` [baseline 逐日 sidecar 诊断产物] 记录 `paper_model_id` [论文模型标识]、`include_count` [include 数量]、`exclude_count` [exclude 数量]、`neutral_count` [neutral 数量]、`selected_asset_count` [原始 include 资产数量]、`optimizer_asset_count` [优化器候选资产数量]、`optimizer_name` [优化器名称]、`optimizer_status` [优化器状态]、`fallback_reason` [fallback 原因]。
    - `baseline_comparison.csv` [baseline 对比表] 区分 DQN signal 环节与 optimizer allocation 环节。
  - 异常兜底：
    - `include` [纳入] 资产少于 2 且可用资产不少于 2：从所有 `非 include 且可用资产` [未被 DQN 判为 include 但仍可交易的资产] 中按 include 优势分数补 top-k 到 2。
    - 可用资产只有 1 个：允许 100% 投该资产，并记录 fallback。
    - 无可用资产：标记 `failed_no_valid_action`。
    - optimizer 失败：fallback 到候选池等权；候选池为空时 fallback 到全部可用资产等权。
    - 无 finite `validation_penalized_sharpe` [验证集惩罚夏普业务指标]：标记 `failed_no_finite_validation_metric`。

- validation 与 checkpoint
  - 输入：`BacktestEngine` [统一回测执行器] validation split daily net returns、daily NAV、turnover 指标。
  - 规则：
    - 两类新增 baseline 的需求/论文别名统一为 `validation_penalized_sharpe` [验证集惩罚夏普业务指标]。
    - 配置层和代码层 canonical key 固定使用 `validation_sharpe_minus_drawdown_turnover_penalty` [净夏普减回撤减换手惩罚指标]。
    - alias 映射固定为 `validation_penalized_sharpe -> validation_sharpe_minus_drawdown_turnover_penalty` [需求/论文别名到配置/代码正式键的映射]。
    - `validation_sharpe_minus_drawdown_turnover_penalty` [净夏普减回撤减换手惩罚指标] 采用平台正式 HPO 指标语义：基于 validation split 的 `daily_returns.net_return` [日度净收益]、`daily_returns.nav` [日度净值] 与 `metrics.average_turnover` [平均换手] 计算，不使用训练 reward，不新增 baseline 专属指标。
    - 当前代码中的 `src/experiments/pipeline.py::objective_metric` [平台 HPO 指标计算函数] 已有 `validation_sharpe_minus_drawdown_turnover_penalty` [净夏普减回撤减换手惩罚指标] 语义：`net_return_sharpe - max_drawdown(nav) - average_turnover`。
    - best checkpoint 必须由 validation metric 选择；final test 前必须加载 best checkpoint；`evaluated_checkpoint_path` [实际评估 checkpoint 路径] 必须写入训练摘要与对比表。
  - 输出：`checkpoint_best_path` [最佳 checkpoint 路径]、`checkpoint_last_path` [最后 checkpoint 路径]、`evaluated_checkpoint_path` [实际评估 checkpoint 路径]、`best_validation_metric` [最佳验证指标]。
  - 异常兜底：checkpoint 缺失、加载失败或 validation metric 非有限时，不得进入 test backtest。

- HPO 与正式主表
  - 输入：`src/experiments/run_experiment.py::NATIVE_HPO_MODEL_NAMES` [native_only HPO 允许名单]、`hpo.native_only` [只允许 native 模型进入 HPO 的配置开关]、`hpo.trainable_models` [HPO 可训练模型列表]。
  - 规则：
    - `hpo.native_only` [只允许 native 模型进入 HPO 的配置开关] 的过滤对象是六个可训练条目，不是两个算法族标识。
    - 每个 seed 内，每个正式 trainable entry 获得相同 `hpo_trials` [HPO trial 数]、`env_steps` [环境步数]、`gradient_updates` [梯度更新次数]、`validation_metric` [验证指标] 与 `selection_rule` [选择规则]。
    - `hybrid_dqn_optimizer_reimplementation` [DQN 信号加组合优化器算法族标识] 不作为 HPO budget 单元。
    - shared-DQN + multi-optimizer final test 只能作为 `diagnostic_shared_dqn` [诊断共享 DQN 方案]，不得进入正式主可比表。
  - 输出：`hpo_model_final_comparison.csv` [HPO 最终模型对比表]、per-model final daily outputs、per-model final comparison metadata。
  - 异常兜底：正式 / HPO / rankable run 中任一必需 child 失败，顶层 `status=failed` [失败状态]，CLI 失败退出。

- 输出产物与 paper aggregation
  - 输入：`src/utils/logger.py` [实验产物写出模块]、`src/experiments/paper_aggregate.py::_paper_model_id` [论文聚合模型身份解析器]、`BacktestEngine` [统一回测执行器] daily outputs。
  - 规则：
    - 不强制扩展 `src/envs/backtest_engine.py::DAILY_REBALANCE_COLUMNS` [daily_rebalance 固定列集合]。
    - 新增逐日诊断写入 `metrics/baseline_daily_diagnostics.csv` [baseline 逐日 sidecar 诊断产物]，按 `date` [估值日期]、`decision_date` [决策日期]、`execution_date` [执行日期]、`model_name` [模型运行标识]、`paper_model_id` [论文模型标识]、`seed` [随机种子]、`fold_id` [fold 标识] 对齐。
    - HPO per-model final 逐日诊断的规范聚合产物为 `metrics/hpo_model_final_daily_diagnostics.csv` [HPO 最终模型逐日诊断产物]；若实现内部生成 per-model sidecar [单模型诊断旁路产物]，必须汇总到该规范聚合产物，且必须保留 `paper_model_id` [论文模型标识]。
    - `baseline_comparison.csv` [baseline 对比表] 写 summary 字段。
    - `baseline_training_history.csv` [训练历史表] 写训练期统计。
    - `daily_rebalance.csv` [逐日调仓产物] 保持兼容。
    - `paper_main_comparison.csv` [论文主比较表] 不保留 `hybrid_dqn_optimizer_reimplementation` [DQN 信号加组合优化器算法族标识] 总行；正式主表只保留五个 `hybrid_dqn_optimizer_*` [optimizer 子模型] 行。
  - 输出：`baseline_comparison.csv` [baseline 对比表]、`baseline_training_summary.csv` [训练摘要]、`baseline_training_history.csv` [训练历史]、`metrics/baseline_daily_diagnostics.csv` [baseline 逐日 sidecar 诊断产物]、`metrics/hpo_model_final_daily_diagnostics.csv` [HPO 最终模型逐日诊断产物]、`daily_returns.csv` [日度收益]、`daily_weights.csv` [日度权重]、`daily_turnover.csv` [日度换手]、`daily_costs.csv` [日度成本]、`daily_rebalance.csv` [日度调仓]、paper-level CSV。
  - 异常兜底：`rankable_in_unified_table=false` [不可进入统一主表]、`proxy_training=true` [代理训练]、`external_original_implementation=true` [外部原始实现] 或 `diagnostic_status=partial_diagnostic` [部分诊断状态] 的行不得进入主排名。

- smoke / diagnostic 状态
  - 输入：显式 smoke / diagnostic run 标记。
  - 规则：
    - `partial_diagnostic` [部分诊断状态] 不作为顶层 `status` [运行状态]。
    - 显式 smoke / diagnostic run 可顶层 `status=completed` [完成状态]，但必须写 `diagnostic_status=partial_diagnostic` [部分诊断状态] 与 `rankable_in_unified_table=false` [不可进入统一主表]。
  - 输出：诊断产物可进入 diagnostic table 或 appendix。
  - 异常兜底：诊断结果不得进入 `paper_main_comparison.csv` [论文主比较表] 主排名。

- 测试边界
  - 输入：新增 baseline 注册、HPO 配置、baseline daily diagnostics、paper aggregation 与 PPO-DQN / Hybrid DQN 行为样例。
  - 规则：测试必须覆盖以下边界：
    - `非 include 且可用资产` [未被 DQN 判为 include 但仍可交易的资产] 可参与候选补足，不限于 `neutral` [中性资产信号]。
    - `validation_penalized_sharpe -> validation_sharpe_minus_drawdown_turnover_penalty` [需求/论文别名到配置/代码正式键的映射]。
    - 六个正式可训练条目在 `DEEP_BASELINE_CLASSES` [baseline 名称到策略类的注册表] 中可被发现。
    - `hpo.native_only` [只允许 native 模型进入 HPO 的配置开关] 保留六个正式可训练条目。
    - `paper_main_comparison.csv` [论文主比较表] 排除 `hybrid_dqn_optimizer_reimplementation` [DQN 信号加组合优化器算法族标识] 总行。
    - `metrics/baseline_daily_diagnostics.csv` [baseline 逐日 sidecar 诊断产物] 与 HPO final diagnostics sidecar 必须落盘，且包含 `paper_model_id` [论文模型标识]。
    - PPO-DQN blend 动作在逐日诊断、`baseline_training_summary.csv` [baseline 训练摘要表] 与 `baseline_comparison.csv` [baseline 对比表] 中记录 `platform_adapted_surrogate=true` [平台适配 surrogate 标记]。
  - 输出：相关单元测试、配置测试、HPO 过滤测试、paper aggregation 测试与诊断产物落盘测试通过。
  - 异常兜底：任一测试边界失败时，不得判定需求实现完成。

- 非目标 / 不改动范围
  - 不复现原论文原始数据、资产池、交易频率、成本假设、测试区间或报告数值。
  - 不使用原论文作者代码作为正式 rankable 结果来源。
  - 不把 `hybrid_dqn_optimizer_reimplementation` [DQN 信号加组合优化器算法族标识] 作为 rankable `paper_model_id` [论文模型标识] 行。
  - 不把 PPO-DQN 的 `hierarchy_action` [PPO-DQN 高层动作] 混同为本项目主模型的 hold/rebalance gate。
  - 不扩展 `daily_rebalance.csv` [逐日调仓产物] 固定列作为本需求的强制路径。
  - 不新增 baseline 专属 validation metric。
  - 不允许 test split 参与模型选择、调参或 checkpoint 选择。
  - 不允许 execution-only 字段进入 observation、DQN signal、optimizer decision window 或 PPO decision input。
